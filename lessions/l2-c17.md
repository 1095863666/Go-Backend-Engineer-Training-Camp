# Error 错误处理深入——为什么 Go 不使用异常，却能构建可靠系统？

------

> **一句话总结本章**

> **Go 把错误作为普通值返回，通过显式处理错误路径，让程序行为更加可预测。**

# 第一节：什么是 error？

先看：

```
err := errors.New("failed")
```

很多人理解：

error 是一个错误对象。

其实：

不是。

------

Go 中：

error 是接口。

定义：

```
type error interface {

    Error() string

}
```

------

也就是说：

只要实现：

```
Error() string
```

就是 error。

------

例如：

```
type MyError struct {

}


func (e MyError) Error() string {

    return "something wrong"

}
```

------

MyError：

就是一个 error。

------

# 第二节：为什么 error 是 interface？

因为：

错误有很多类型。

例如：

数据库错误：

```
DatabaseError
```

------

网络错误：

```
NetworkError
```

------

权限错误：

```
PermissionError
```

------

如果固定：

一种类型。

扩展困难。

------

接口：

提供：

统一抽象。

------

# 第三节：Go 为什么不用异常？

先看 Java：

```
try {

    userService.save();

}

catch(Exception e){

}
```

------

问题：

调用者：

不知道：

哪里可能失败。

------

例如：

```
save()
```

可能：

抛：

- DBException
- NetworkException
- PermissionException

------

但是：

方法签名：

看不出来。

------

# 第四节：Go 的设计

Go：

直接：

返回。

例如：

```
user, err := GetUser(id)
```

------

函数签名：

告诉你：

可能失败。

------

这叫：

显式错误。

------

# 第五节：最基础错误创建

## errors.New

例如：

```
import "errors"


err := errors.New("user not found")
```

------

返回：

一个 error。

------

适合：

固定错误。

------

例如：

```
var ErrNotFound = errors.New("not found")
```

------

# 第六节：为什么定义全局错误？

例如：

用户查询：

很多地方。

------

不要：

每次：

```
errors.New("not found")
```

------

因为：

每次：

创建不同对象。

------

推荐：

```
var ErrUserNotFound = errors.New("user not found")
```

------

然后：

判断：

```
if err == ErrUserNotFound {

}
```

------

# 第七节：错误比较的问题

例如：

```
err1 := errors.New("failed")

err2 := errors.New("failed")
```

------

比较：

```
err1 == err2
```

结果：

false。

------

为什么？

因为：

两个对象。

------

所以：

不要：

直接比较动态创建的 error。

------

# 第八节：fmt.Errorf

如果需要：

动态信息。

例如：

用户 ID。

------

不要：

```
errors.New(
"user 10086 not found"
)
```

------

使用：

```
fmt.Errorf(
"user %d not found",
id,
)
```

------

例如：

```
err := fmt.Errorf(
"user %d not found",
100,
)
```

------

结果：

```
user 100 not found
```

------

# 第九节：错误包装（Wrapping）

这是 Go 1.13：

重要特性。

------

场景：

数据库：

返回：

```
sql.ErrNoRows
```

------

业务层：

需要增加信息。

------

以前：

```
return err
```

信息丢失。

------

现在：

```
return fmt.Errorf(
"query user failed: %w",
err,
)
```

------

`%w`

表示：

包装原错误。

------

# 第十节：为什么需要包装？

例如：

调用链：

```
API

 ↓

Service

 ↓

Repository

 ↓

Database
```

------

数据库：

错误：

```
record not found
```

------

Service：

增加：

```
get user failed
```

------

API：

增加：

```
request failed
```

------

最终：

错误：

包含：

完整上下文。

------

# 第十一节：errors.Is

问题：

包装以后：

还能判断吗？

------

例如：

```
return fmt.Errorf(
"user error: %w",
ErrNotFound,
)
```

------

现在：

不能：

```
err == ErrNotFound
```

------

因为：

外面包了一层。

------

使用：

```
errors.Is(err, ErrNotFound)
```

------

作用：

检查：

错误链。

------

# 第十二节：errors.As

除了判断：

有时候：

需要拿到具体错误。

------

例如：

定义：

```
type ValidationError struct {

Field string

}
```

------

实现：

error。

------

然后：

```
var ve *ValidationError


if errors.As(err,&ve){

}
```

------

可以：

获取：

详细信息。

------

# 第十三节：error chain

现在：

错误：

不是一个值。

而是一条链。

------

例如：

```
API Error

↓

Service Error

↓

Repository Error

↓

SQL Error
```

------

errors.Is：

沿链寻找。

------

errors.As：

沿链提取。

------

# 第十四节：企业错误设计

真实项目：

不会：

随便：

errors.New。

------

通常：

分层。

------

例如：

domain：

```
var (
    ErrNotFound = errors.New("not found")
    ErrForbidden = errors.New("forbidden")
)
```

------

service：

包装。

------

api：

转换：

HTTP 状态码。

------

# 第十五节：HTTP API 错误设计

例如：

业务：

```
ErrUserNotFound
```

------

API：

转换：

```
{
 "code":10001,
 "message":"user not found"
}
```

------

HTTP：

```
404
```

------

不要：

直接返回：

内部错误。

------

错误：

可能泄露：

数据库信息。

# 第十六节：Flow Engine 错误设计

流程执行：

一定会失败。

例如：

节点：

```
SQL Node
```

失败：

原因：

- SQL错误
- 超时
- 参数错误
- 权限错误

------

设计：

```
type NodeError struct {

    NodeID string

    Reason error

}
```

------

实现：

error。

------

执行：

```
func Execute() error {

}
```

------

返回：

```
return &NodeError{

    NodeID:"node1",

    Reason:err,

}
```

------

# 第十七节：为什么 Flow Engine 需要错误分类？

因为：

失败后：

处理不同。

------

例如：

## 用户输入错误

状态：

FAILED。

无需重试。

------

## 网络错误

状态：

RETRY。

可以重新执行。

------

## 系统错误

状态：

PAUSED。

等待人工。

------

所以：

错误类型：

影响：

流程状态机。

------

# 第十八节：错误处理最佳实践

------

## 1

不要忽略 error。

错误：

```
data,_ := Read()
```

危险。

------

## 2

增加上下文。

不要：

```
return err
```

------

推荐：

```
return fmt.Errorf(
"save user failed: %w",
err,
)
```

------

## 3

不要用字符串判断。

错误：

```
if err.Error()=="not found"
```

------

使用：

```
errors.Is()
```

------

## 4

错误应该服务业务。

------

# 第二十节：panic 和 error 区别

error：

可恢复。

例如：

用户不存在。

------

panic：

程序无法继续。

例如：

程序初始化失败。

------

例如：

配置文件不存在：

启动阶段：

panic 可以。

------

运行请求：

不要 panic。
