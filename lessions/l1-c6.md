# 为什么 Go 成为了云原生时代的语言？

------

> **一句话总结本章**

> **Go 的成功不是因为它替代了所有语言，而是因为它刚好满足了云时代基础设施软件的核心需求。**

------

# 本章学习目标

完成本章以后，你应该理解：

- 什么是云原生？
- 云时代的软件和传统软件有什么区别？
- 为什么基础设施喜欢 Go？
- Docker 为什么使用 Go？
- Kubernetes 为什么使用 Go？
- Go 和微服务为什么契合？
- Go 在企业中的真实定位是什么？

------

# 第一节：什么是云原生？

在理解 Go 之前。

先理解：

什么叫：

Cloud Native。

中文：

云原生。

------

很多人认为：

云原生就是：

"把程序部署到云服务器。"

这是错误的。

------

云原生真正的意思：

> **软件从设计开始，就为了云环境而生。**

------

传统软件：

像这样：

```
服务器

↓

安装程序

↓

运行

↓

几年不变
```

------

云原生：

像这样：

```
用户请求

↓

服务A

↓

服务B

↓

服务C

↓

容器

↓

自动扩容

↓

自动恢复
```

------

特点：

- 服务很多
- 快速部署
- 自动扩缩容
- 高并发
- 分布式
- 自动管理

------

# 第二节：互联网系统发生了变化

过去：

一个大型系统：

可能：

```
一个应用

+

一个数据库
```

例如：

传统 ERP。

------

结构：

```
Browser

↓

Java Application

↓

Database
```

------

但是今天：

大型互联网系统：

更像：

```
                 Gateway

                    |

    ┌───────────────┼───────────────┐

    ↓               ↓               ↓

 User Service   Order Service   Payment Service


    ↓               ↓               ↓


 Database        Cache          MQ
```

------

这就是：

微服务。

------

# 第三节：微服务需要什么语言？

我们分析一下。

一个微服务：

通常特点：

------

## 第一：小

一个服务：

可能：

只负责一个功能。

例如：

订单服务。

------

## 第二：数量多

不是：

10 个。

可能：

100 个。

1000 个。

------

## 第三：快速部署

今天：

修改代码。

马上：

上线。

------

## 第四：长期运行

服务器：

一年：

365 天运行。

------

所以：

需要：

一种语言：

- 编译快
- 启动快
- 内存少
- 并发强
- 部署简单

------

你会发现：

这正是：

Go 的目标。

------

# 第四节：为什么 Docker 使用 Go？

Docker 是云时代的重要转折点。

------

Docker 解决什么？

以前：

开发：

"我电脑能运行。"

上线：

"服务器不能运行。"

------

原因：

环境不同。

------

Docker 提出：

容器。

------

一个应用：

包括：

```
代码

+

依赖

+

运行环境
```

打包。

------

为什么 Docker 选择 Go？

原因：

## 1. 单文件部署

Docker 本身：

就是一个命令行工具。

例如：

```
docker run nginx
```

------

Go：

编译：

```
docker
```

一个二进制文件。

非常适合。

------

## 2. 系统编程能力

Docker 需要：

操作：

- Linux namespace
- cgroup
- 网络
- 文件系统

Go 可以很好地调用系统能力。

------

## 3. 并发

Docker daemon：

需要：

同时管理：

- 容器
- 网络
- 镜像
- 日志

大量并发任务。

Go 很适合。

------

# 第五节：为什么 Kubernetes 使用 Go？

如果说：

Docker 开启容器时代。

那么：

Kubernetes 管理容器时代。

------

Kubernetes 做什么？

简单理解：

它管理：

大量服务器上的大量容器。

------

例如：

你有：

1000 个服务。

Kubernetes 负责：

- 创建
- 删除
- 调度
- 扩容
- 恢复

------

它内部：

大量控制循环。

类似：

```
while true:

    检查状态

    如果不正确:

        修复
```

------

这种系统：

需要：

大量并发。

大量网络通信。

长期运行。

------

Go 非常适合。

------

# 第六节：Go 和并发模型

这是一个关键原因。

传统语言：

并发：

通常：

线程。

------

线程：

操作系统管理。

成本较高。

------

Go：

引入：

goroutine。

简单理解：

一种非常轻量的任务。

------

例如：

服务器收到请求：

传统：

```
请求1

创建线程


请求2

创建线程


请求3

创建线程
```

------

Go：

```
请求1

goroutine


请求2

goroutine


请求3

goroutine
```

------

一个 Go 程序：

可以运行：

几十万 goroutine。

------

这非常符合：

互联网服务模型。

------

# 第七节：为什么 Go 适合基础设施？

现在总结一下。

基础设施软件：

有什么特点？

------

## 特点1：

没有复杂 UI。

例如：

Kubernetes。

没有漂亮页面。

核心：

后台服务。

------

Go：

非常适合。

------

## 特点2：

网络密集。

大量：

HTTP。

RPC。

TCP。

------

Go：

标准库：

网络支持优秀。

------

## 特点3：

长期运行。

服务器：

不停运行。

------

Go：

部署简单。

稳定。

------

## 特点4：

团队维护。

很多人参与。

------

Go：

代码统一。

容易阅读。

------

# 第八节：Go 和 Java 的关系

这里需要澄清。

很多人认为：

Go 出现：

就是为了干掉 Java。

不是。

------

实际上：

两个定位不同。

------

Java：

强项：

- 企业系统
- 大型业务
- ERP
- 金融系统
- 大量成熟框架

------

Go：

强项：

- 基础设施
- 云服务
- 网络服务
- DevOps
- 微服务

------

例如：

一个银行：

可能：

核心交易：

Java。

基础设施：

Go。

------

不是竞争。

是选择。

------

# 第九节：Go 和 Python 的关系

同样。

Go：

不是替代 Python。

------

Python：

强项：

- AI
- 数据分析
- 科学计算
- 自动化

------

Go：

强项：

- 服务
- 网络
- 基础设施

------

现实：

很多 AI 公司：

组合使用。

例如：

```
Python

负责模型训练


Go

负责服务部署
```

------

这是一种工程现实。

------

# 第十节：为什么 Flow Engine 适合 Go？

回到你的毕业项目。

Flow Engine。

为什么适合 Go？

分析。

------

流程引擎特点：

## 1. 节点执行

大量任务。

需要：

并发。

Go：

优势。

------

## 2. 长时间运行

流程：

可能：

执行几个小时。

甚至几天。

Go：

后台服务能力强。

------

## 3. 状态管理

需要：

- Context
- 并发控制
- 任务调度

Go：

生态成熟。

------

## 4. 企业长期维护

流程系统：

生命周期：

可能：

十年以上。

Go：

强调：

可维护性。

------

所以：

Flow Engine。

非常适合作为：

Go 工程训练项目。
