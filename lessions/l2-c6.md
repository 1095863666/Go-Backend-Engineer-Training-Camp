# Array 与 Slice（一）——为什么 Go 发明了 Slice？

------

> **一句话总结本章**

> **Slice 不是数组，它是 Go 为了解决数组不灵活而设计的一层"视图（View）"。**

# 第一节：为什么程序需要存储多个数据？

前面我们学习了变量。

例如：

```
score := 98
```

没有问题。

但是。

如果现在：

一个班：

50 名学生。

怎么办？

你会不会这样写？

```
score1 := 98
score2 := 87
score3 := 92
...
score50 := 66
```

当然不会。

程序需要：

**一次管理一组相同类型的数据。**

于是。

数组（Array）出现了。

------

# 第二节：什么是数组？

来看 Go。

```
var scores [5]int
```

很多教程直接告诉你：

> 定义了一个长度为 5 的数组。

但是我们继续往下问。

**为什么长度必须写出来？**

答案：

因为数组本质上就是：

> **一块连续的内存。**

例如：

```
scores := [5]int{10,20,30,40,50}
```

内存可能像这样：

```
地址

0x1000  -> 10

0x1008  -> 20

0x1016  -> 30

0x1024  -> 40

0x1032  -> 50
```

注意：

它们：

**连续排列。**

为什么连续？

因为：

CPU 最喜欢连续内存。

------

# 第三节：连续内存为什么快？

假设：

CPU 要读取：

第一个元素。

由于连续。

它可以：

顺便把后面的数据一起加载。

这叫：

**Cache Locality（缓存局部性）。**

例如：

```
CPU Cache

↓

10

20

30

40
```

下一次访问：

20。

根本不用访问内存。

速度非常快。

------

所以。

数组最大的优点：

就是：

**连续。**

------

# 第四节：数组最大的缺点

假设：

数组：

```
scores := [5]int{}
```

后来：

需要：

第六个元素。

怎么办？

没有地方。

因为：

数组：

创建以后：

大小固定。

例如：

```
10

20

30

40

50
```

你想：

加入：

60。

必须：

重新创建：

```
10

20

30

40

50

60
```

然后：

复制。

成本很高。

------

所以。

数组：

适合：

固定数量。

例如：

```
一年

12 个月

↓

固定
```

例如：

星期：

```
7 天
```

例如：

IPv4：

```
4 个字节
```

这些都适合数组。

------

# 第五节：现实世界的数据很少固定

来看一个网站。

用户数量：

固定吗？

不是。

订单数量：

固定吗？

不是。

日志：

固定吗？

不是。

消息：

固定吗？

不是。

几乎所有业务数据：

都是：

**动态增长。**

如果全部：

数组。

程序每天：

都在复制内存。

效率很低。

------

于是。

Go 设计了：

**Slice。**

------

# 第六节：为什么叫 Slice？

英文：

Slice。

很多人翻译：

切片。

其实：

这个翻译容易误导。

Slice 原来的意思：

是：

> **从一个整体切下来的一部分。**

例如：

披萨。

```
整个披萨

↓

切一块

↓

Slice
```

例如：

蛋糕。

```
Cake

↓

Slice
```

所以。

Go 的 Slice。

最初的设计思想就是：

> **从数组切下来的一段。**

------

# 第七节：第一个 Slice

例如：

```
arr := [5]int{10,20,30,40,50}
```

现在：

```
s := arr[1:4]
```

很多教程直接告诉你：

结果：

```
20

30

40
```

但是：

真正发生的是：

```
数组

10

20

30

40

50

       ↑

      Slice
```

注意。

Slice：

没有复制数据。

它只是：

**指向数组的一部分。**

------

# 第八节：Slice 不是数组

这是 Go 面试第一高频。

很多新人认为：

```
[]int
```

就是：

动态数组。

不是。

Go 官方文档明确说：

> Slice 是对数组的一段描述（descriptor）。

什么意思？

Slice：

本身：

不是数据。

而是：

一个说明书。

里面记录：

```
数组地址

长度

容量
```

真正的数据：

还在数组。

------

# 第九节：Slice 到底长什么样？

编译器内部。

一个 Slice。

可以理解成：

```
type slice struct{

    ptr *int

    len int

    cap int

}
```

注意。

这是：

理解模型。

不是源码。

里面：

三个成员。

------

## ptr

指向数组。

------

## len

当前长度。

------

## cap

还能扩展多少。

------

所以。

Slice：

本质：

只是：

三个数字。

真正的数据：

不在 Slice 里面。

------

# 第十节：为什么这样设计？

假设。

Slice 真的是：

复制数组。

例如：

100 MB。

那么：

函数传参：

```
func Process(data []byte)
```

每次调用：

复制：

100 MB。

服务器：

直接崩。

------

现在：

Slice：

只有：

```
ptr

len

cap
```

几十个字节。

复制成本：

极低。

这就是：

Go 的设计。

------

# 第十一节：Slice 和 Python List 的区别

很多人：

学过 Python。

容易误解。

Python：

```
a=[1,2,3]
```

List：

就是：

真正的数据结构。

里面管理：

元素。

容量。

扩容。

全部自己完成。

------

Go：

不是。

Go：

数据：

一直：

在数组。

Slice：

只是：

窗口。

例如：

```
数组：

10

20

30

40

50

Slice：

    20

    30

    40
```

以后。

多个 Slice：

还能：

共享：

同一个数组。

这也是很多 Bug 的来源。

------

# 第十二节：企业中的 Slice

例如：

数据库：

查询：

```
users := []User{}
```

为什么：

不是：

数组？

因为：

不知道：

多少用户。

------

HTTP：

请求：

```
headers := []Header{}
```

不知道：

多少 Header。

------

流程引擎：

执行节点：

```
nodes := []Node{}
```

节点数量：

动态变化。

所以。

几乎所有 Go 项目。

大量使用：

Slice。

而不是：

数组。

------
