# Go 到底解决了什么问题？

------

> **一句话总结本章**

> **Go 解决的不是"如何写一个程序"，而是"如何让几千名工程师长期维护一个巨大的程序系统"。**

------

# 本章学习目标

完成本章后，你应该能够理解：

- 为什么大型项目会越来越难维护？
- 为什么代码复杂度会随着团队扩大而爆炸？
- 为什么 Go 强调简单？
- 为什么 Go 强调组合？
- 为什么 Go 强调并发？
- 为什么 Go 强调快速编译？
- 为什么 Go 最终成为云原生时代的重要语言？

------

# 第一节：一个错误的问题

很多初学者学习 Go 时，会问：

> Go 比 C++ 快吗？

或者：

> Go 比 Java 简单吗？

这些问题没有错。

但是：

不是 Go 诞生的真正原因。

Go 诞生的问题是：

------

## 如果一个项目存在 10 年，会发生什么？

假设：

今天。

你写了一个项目：

```
project

10000 行代码
```

你觉得：

非常简单。

------

一年以后：

```
project

50000 行代码
```

来了：

5 个工程师。

------

三年以后：

```
project

500000 行代码
```

来了：

50 个工程师。

------

十年以后：

```
project

5000000 行代码
```

已经：

200 人维护。

------

这时候。

最大的成本是什么？

不是 CPU。

不是内存。

而是：

------

# 理解代码的成本

------

有人说：

> 写代码很快，读代码很慢。

这是软件工程里面非常重要的一句话。

实际开发中：

很多时间不是写新功能。

而是：

```
阅读旧代码

↓

理解设计

↓

找到问题

↓

修改

↓

避免破坏其他地方
```

------

所以：

软件生命周期：

大部分成本：

发生在维护阶段。

------

# 第二节：复杂度如何爆炸？

假设：

一个项目只有一个人。

他写：

```
func calculate()
```

他知道：

这个函数做什么。

------

后来：

5 个人。

出现：

```
calculate()

validate()

process()

handle()
```

大家需要沟通。

------

后来：

100 个人。

关系变成：

```
A 修改函数

↓

影响 B 模块

↓

B 修改接口

↓

影响 C 服务

↓

C 修改数据库
```

复杂度开始增长。

------

这就是：

软件工程里的经典问题：

## Communication Cost

沟通成本。

------

# 第三节：Brooks 定律

软件工程有一个非常著名的规律：

> Adding manpower to a late software project makes it later.

中文：

> 给一个已经延期的软件项目增加人手，往往会让项目更延期。

为什么？

因为新人需要：

学习。

沟通。

理解。

培训。

------

举个简单例子。

一个人：

```
知道所有代码
```

两个人：

需要：

```
A ↔ B
```

沟通。

三个人：

```
A ↔ B

A ↔ C

B ↔ C
```

六条关系。

十个人：

关系数量：

接近：

45。

------

人数增加。

沟通成本快速增加。

------

所以：

大型团队最需要什么？

不是：

更多功能。

而是：

降低沟通成本。

------

# 第四节：Go 的第一个答案 —— 简单

这就是为什么：

Go 喜欢简单。

------

例如：

C++：

允许：

```
多继承

模板元编程

操作符重载

宏
```

这些能力：

很强。

但是：

也意味着：

同一个问题。

可能：

有很多写法。

------

例如：

一个对象创建。

可能：

```
普通构造

工厂模式

模板工厂

单例

依赖注入

宏生成
```

新人看到：

不知道：

作者为什么选择这个。

------

Go 的想法：

减少选择。

------

# 第五节：为什么 Go 没有继承？

很多人第一次学习 Go：

都会问：

> 为什么没有 class？

> 为什么没有 extends？

这是非常好的问题。

------

传统面向对象：

通常：

```
Animal

   ↑

Dog
```

Dog：

继承 Animal。

------

Go：

更喜欢：

组合。

例如：

```
type Animal struct {
    Name string
}


type Dog struct {
    Animal
}
```

意思：

Dog：

拥有：

Animal。

而不是：

Dog 是一种 Animal。

------

为什么？

因为大型系统里面：

继承关系容易变复杂。

例如：

```
A

↓

B

↓

C

↓

D

↓

E
```

修改 A。

可能影响：

整个链。

------

组合：

更加明确。

------

# 第六节：Go 的第二个答案 —— 明确

大型项目最大的问题：

不是不会写。

而是不知道：

代码到底做什么。

------

所以 Go 喜欢：

明确。

------

例如错误处理。

很多语言：

```
try {

    doSomething();

}
catch(Exception e){

}
```

错误：

跳到另外地方。

------

Go：

```
result, err := doSomething()

if err != nil {
    return err
}
```

错误：

就在这里处理。

------

有人觉得：

重复。

但是 Go 的观点：

> 重复一点代码，比隐藏控制流程更容易维护。

------

# 第七节：Go 的第三个答案 —— 并发

2007 年。

互联网已经进入：

高并发时代。

服务器面对：

```
100

1000

10000

100000

连接
```

------

传统方式：

一个线程一个请求。

问题：

线程成本高。

------

例如：

100 万请求。

如果：

100 万线程。

显然：

不现实。

------

所以需要：

轻量级并发。

------

Go 提出：

goroutine。

以后详细学习。

这里先理解：

它希望：

让并发成为语言的一部分。

------

不是：

额外技术。

------

# 第八节：Go 的第四个答案 —— 快速部署

现代互联网：

服务很多。

例如：

一个系统：

```
用户服务

订单服务

支付服务

库存服务

消息服务
```

------

部署：

越简单越好。

------

Go：

编译：

生成：

```
app
```

一个文件。

运行：

```
./app
```

即可。

------

不需要：

JVM。

不需要：

解释器。

------

这对于：

微服务。

非常重要。

------

# 第九节：为什么 Go 后来成为云原生语言？

现在看：

Docker。

Kubernetes。

Prometheus。

Terraform。

为什么大量使用 Go？

因为它们面对的问题：

和 Google 当年的问题非常类似：

------

## Docker

需要：

- 快
- 易部署
- 网络
- 并发

Go 很合适。

------

## Kubernetes

管理：

大量服务器。

需要：

- API
- 并发
- 分布式
- 工具链

Go 很合适。

------

## Prometheus

监控系统。

需要：

- 长时间运行
- 高并发采集
- 资源效率

Go 很合适。

------

Go 不是：

最强大的语言。

不是：

最快的语言。

不是：

功能最多的语言。

Go 是：

```
适合大型团队

+

长期维护

+

服务器程序

+

并发系统

+

云基础设施
```

的一门工程语言。
