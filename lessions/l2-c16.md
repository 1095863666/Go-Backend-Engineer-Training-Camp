# Go Module 与依赖管理——为什么 Go 放弃 GOPATH，拥抱模块化？

------

> **一句话总结本章**

> **Go Module 通过 go.mod 和 go.sum 管理项目身份与依赖版本，让 Go 项目脱离固定目录，实现可靠构建。**

------

# 本章学习目标

完成本章后，你应该能够：

- 理解 GOPATH 的历史问题
- 理解 Go Module 出现原因
- 理解 go.mod
- 理解 go.sum
- 理解依赖版本管理
- 理解 replace
- 理解私有 module
- 能管理企业 Go 项目依赖

------

# 第一节：Go 早期为什么需要 GOPATH？

早期 Go：

没有 Module。

采用：

GOPATH。

------

什么是 GOPATH？

简单理解：

一个固定工作目录。

例如：

```
id="q3k9va"
GOPATH

└── src

    └── github.com

        └── company

            └── project
```

------

所有代码：

必须放这里。

------

# 第二节：GOPATH 的问题

听起来：

简单。

但是：

大型项目：

马上遇到问题。

------

## 问题 1：项目位置固定

例如：

公司项目：

必须：

放：

```
$GOPATH/src
```

------

不能：

随便：

放桌面。

------

# 问题 2：版本无法隔离

例如：

项目 A：

需要：

```
gin v1.8
```

项目 B：

需要：

```
gin v1.10
```

------

GOPATH：

只有：

一个版本。

------

冲突。

------

# 问题 3：无法复现环境

开发：

机器 A：

能运行。

机器 B：

失败。

------

因为：

依赖：

不明确。

------

所以：

Go 需要：

模块化。

------

# 第三节：Go Module 出现

Go 1.11：

引入：

Go Module。

------

核心思想：

> 每个项目自己管理依赖。

------

项目：

拥有：

```
go.mod
```

------

里面：

记录：

需要什么。

------

# 第四节：创建 Module

例如：

新项目：

```
go mod init github.com/company/flow-engine
```

------

生成：

```
id="i0o8up"
flow-engine

├── go.mod

└── main.go
```

------

go.mod：

```
id="k3p9bv"
module github.com/company/flow-engine


go 1.22
```

------

# 第五节：module 名是什么？

例如：

```
id="1d4s2k"
module github.com/company/flow-engine
```

------

它表示：

项目唯一身份。

------

以后：

别人：

import：

```
id="g3l9k4"
github.com/company/flow-engine/user
```

------

找到：

你的 package。

------

# 第六节：添加依赖

例如：

使用 Gin。

代码：

```
id="5p7z6x"
import "github.com/gin-gonic/gin"
```

------

执行：

```
go mod tidy
```

------

Go：

自动：

下载。

------

go.mod：

增加：

```
id="p2x8r9"
require (

github.com/gin-gonic/gin v1.9.1

)
```

------

# 第七节：go.mod 作用

go.mod：

主要记录：

三个东西。

------

## 1. 项目身份

例如：

```
module github.com/company/project
```

------

## 2. Go 版本

例如：

```
go 1.22
```

------

## 3. 依赖

例如：

```
require (

gorm.io/gorm v1.25

)
```

------

# 第八节：go.sum 是什么？

很多新人：

看到：

```
go.sum
```

不知道干什么。

------

简单：

go.mod：

记录：

我要什么。

------

go.sum：

记录：

我下载的东西是否可信。

------

类似：

文件指纹。

------

例如：

```
github.com/gin-gonic/gin v1.9.1 h1:xxxx
```

------

防止：

依赖被偷偷修改。

------

# 第九节：为什么需要 go.sum？

假设：

今天：

下载：

gin。

------

明天：

有人：

发布：

同版本：

但是内容不同。

------

如果没有校验：

项目：

可能：

行为变化。

------

go.sum：

保证：

内容一致。

------

# 第十节：go mod tidy

非常重要。

作用：

整理依赖。

------

例如：

你删除：

某个 import。

但是：

go.mod：

还有：

依赖。

------

执行：

```
go mod tidy
```

------

Go：

删除无用依赖。

------

企业：

经常：

执行。

------

# 第十一节：依赖版本规则

Go：

使用：

语义化版本。

例如：

```
v1.2.3
```

表示：

```
主版本.次版本.修订版本
```

------

一般：

升级：

小版本：

风险低。

------

大版本：

可能：

破坏兼容。

------

# 第十二节：版本选择

例如：

```
require github.com/gin-gonic/gin v1.9.1
```

表示：

固定。

------

团队：

通常：

提交：

go.mod。

------

所有人：

使用：

同版本。

------

# 第十三节：replace 是什么？

开发中：

非常常见。

------

例如：

你依赖：

```
github.com/company/common
```

------

但是：

现在：

你想测试：

本地代码。

------

可以：

```
replace github.com/company/common => ../common
```

------

意思：

不要下载。

使用：

本地目录。

------

# 第十四节：Flow Engine 实战

假设：

项目：

```
flow-engine
```

依赖：

Gin。

GORM。

Redis。

------

go.mod：

类似：

```
module github.com/company/flow-engine


go 1.22


require (

github.com/gin-gonic/gin v1.9.1

gorm.io/gorm v1.25.5

github.com/redis/go-redis/v9 v9.5.0

)
```

------

这样：

部署服务器：

只需要：

```
go mod download
```

------

环境一致。

------

# 第十五节：企业私有 Module

公司内部：

例如：

```
github.com/company/common
```

------

项目：

依赖：

```
require github.com/company/common v1.0.0
```

------

但是：

私有仓库：

需要配置。

例如：

```
GOPRIVATE=github.com/company/*
```

------

告诉 Go：

不要去公共代理。

------

# 第十六节：Go Proxy

Go 下载依赖：

通常：

经过：

代理。

------

例如：

默认：

```
proxy.golang.org
```

------

作用：

缓存。

加速。

------

企业：

可能：

使用：

私有代理。

------

因为：

------
