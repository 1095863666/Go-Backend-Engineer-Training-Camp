# Defer、Panic、Recover——Go 如何管理资源和处理不可恢复错误？

------

> **一句话总结本章**

> **defer 用于保证资源释放，panic 用于处理无法继续运行的严重错误，recover 用于恢复 panic，三者共同构成 Go 的异常控制机制。**

------

- 理解 defer 执行顺序
- 理解 defer 参数计算
- 理解 defer 和闭包
- 理解 defer 与返回值
- 理解 panic

- 在实际项目中正确使用它们

# 第一节：为什么需要 defer？

先看传统写法。

例如：

读取文件：

```
func ReadFile(){

    file, err := os.Open("test.txt")

    if err != nil {

        return

    }


    data := make([]byte,100)

    file.Read(data)


    file.Close()

}
```

------

问题：

如果中间：

增加：

```
return
```

或者：

发生错误：

```
panic()
```

那么：

```
file.Close()
```

可能：

不会执行。

------

资源泄漏。

------

# 第二节：defer 是什么？

defer：

意思：

延迟执行。

------

例如：

```
func main(){

    defer fmt.Println("world")


    fmt.Println("hello")

}
```

输出：

```
hello

world
```

------

为什么？

因为：

defer 注册的函数：

会在当前函数结束前执行。

------

# 第三节：defer 最常见用途

资源释放。

例如：

文件：

```
file, err := os.Open("test.txt")

if err != nil {

    return

}


defer file.Close()
```

------

意思：

无论后面发生什么：

函数退出前：

执行：

```
file.Close()
```

------

这就是：

Go 推荐写法。

------

# 第四节：defer 执行时机

注意：

不是立即执行。

例如：

```
func test(){

    defer fmt.Println("A")


    fmt.Println("B")

}
```

执行过程：

进入函数：

注册 defer。

------

执行：

```
B
```

------

函数结束：

执行：

```
A
```

------

结果：

```
B

A
```

------

# 第五节：多个 defer

例如：

```
func test(){

    defer fmt.Println("1")

    defer fmt.Println("2")

    defer fmt.Println("3")

}
```

------

输出：

```
3

2

1
```

------

为什么？

因为：

defer 是：

栈结构。

------

叫：

LIFO。

Last In First Out。

------

后进先出。

------

# 第六节：defer 参数什么时候计算？

这是面试重点。

看：

```
func main(){

    x:=10


    defer fmt.Println(x)


    x=20

}
```

------

输出：

是什么？

很多人认为：

20。

------

实际：

10。

------

为什么？

因为：

defer 注册的时候：

参数已经计算。

------

等价：

```
tmp:=x

defer fmt.Println(tmp)
```

------

# 第七节：defer + 闭包

但是：

如果：

写：

```
func main(){

    x:=10


    defer func(){

        fmt.Println(x)

    }()


    x=20

}
```

------

输出：

```
20
```

------

为什么？

因为：

闭包捕获变量。

不是值。

------

区别：

------

普通参数：

```
defer f(x)
```

立即计算。

------

闭包：

```
defer func(){

}()
```

执行时读取。

------

# 第八节：defer 与返回值

这是高级内容。

看：

```
func test() int{

    x:=10


    defer func(){

        x++

    }()


    return x

}
```

------

返回：

多少？

------

答案：

10。

------

为什么？

return：

不是一步。

------

过程：

类似：

```
计算返回值

↓

保存返回值

↓

执行 defer

↓

返回
```

------

所以：

x 修改：

不会影响。

------

# 第九节：命名返回值

但是：

如果：

```
func test()(x int){

    x=10


    defer func(){

        x++

    }()


    return

}
```

------

结果：

11。

------

为什么？

因为：

x：

就是返回变量。

------

defer：

修改它。

------

# 第十节：defer 在数据库事务中的应用

真实场景：

事务：

```
func CreateOrder(){

    tx.Begin()


    defer tx.Rollback()


    save()


    tx.Commit()

}
```

------

为什么：

先 Rollback？

因为：

如果中间失败：

自动回滚。

------

成功：

Commit。

------

但是：

注意：

实际项目：

需要判断。

------

# 第十一节：panic 是什么？

panic：

表示：

程序进入不可恢复状态。

------

例如：

```
panic("something wrong")
```

------

执行：

立即停止当前流程。

------

开始：

panic 传播。

------

# 第十二节：panic 和 error 区别

error：

业务错误。

例如：

用户不存在。

------

panic：

程序错误。

例如：

数组越界。

------

比较：

| 类型  | 用途         |
| ----- | ------------ |
| error | 正常失败     |
| panic | 程序无法继续 |

------

# 第十三节：panic 传播过程

例如：

```
func A(){

    B()

}


func B(){

    panic("error")

}
```

------

过程：

```
B panic

↓

A 没处理

↓

main

↓

程序退出
```

------

panic：

向上传播。

------

# 第十四节：recover 是什么？

recover：

恢复 panic。

------

只能：

在 defer 中使用。

------

例如：

```
func main(){

    defer func(){

        if err:=recover();err!=nil{

            fmt.Println(err)

        }

    }()


    panic("oops")

}
```

------

输出：

```
oops
```

------

程序：

继续。

------

# 第十五节：为什么 recover 必须在 defer？

因为：

panic：

发生时：

正常代码：

不会继续。

------

例如：

错误：

```
recover()
```

不会执行。

------

必须：

提前注册：

```
defer func(){

recover()

}()
```

------

# 第十六节：HTTP 服务为什么使用 recover？

真实 Web 服务：

不能：

一个请求 panic。

导致：

整个服务挂掉。

------

所以：

Gin 等框架：

通常：

有 Recovery Middleware。

------

流程：

```
Request

↓

Handler panic

↓

recover

↓

返回 500
```

------

# 第十七节：Flow Engine 中 panic 怎么处理？

流程节点：

例如：

```
Execute()
```

------

正常：

返回：

```
error
```

------

但是：

如果插件：

代码错误：

panic。

------

怎么办？

不能：

让整个 Worker 崩溃。

------

设计：

```
func SafeExecute(){

    defer func(){

        if r:=recover(); r!=nil{

            convertToError()

        }

    }()

}
```

------

转换：

panic

↓

NodeError

------

# 第十八节：什么时候使用 panic？

推荐：

非常少。

------

适合：

## 初始化失败

例如：

配置完全错误。

------

## 程序不变量破坏

例如：

理论不可能发生。

------

## 库内部严重错误

------

# 第十九节：什么时候不要 panic？

不要：

业务错误。

例如：

错误：

```
panic("user not found")
```

------

应该：

```
return ErrUserNotFound
```

------

不要：

请求处理。

------

不要：

数据库失败。

------

# 第二十节：defer 性能问题

以前：

有人说：

defer 很慢。

------

现代 Go：

已经优化。

------

普通业务：

不用担心。

------

但是：

高性能循环：

注意。

例如：

错误：

```
for i:=0;i<1000000;i++{

    defer close()

}
```

------

会积累大量 defer。

------

# 第二十一节：企业实践规范

推荐：

------

## 文件

```
file,err:=open()

if err!=nil{

}

defer file.Close()
```

------

## 锁

```
mutex.Lock()

defer mutex.Unlock()
```

------

## 事务

```
tx.Begin()

defer rollback()
```

------

## 网络连接

```
resp.Body.Close()
```

# 第二十二节：Flow Engine 实战

节点执行：

```
func ExecuteNode(node Node) error {

    defer func(){

        recover()

    }()


    err:=node.Execute()


    return err

}
```

------

保证：

节点错误：

不会导致：

整个 Worker 崩溃。
