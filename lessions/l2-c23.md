# 第一节：什么是 Go Runtime？

很多语言：

依赖：

操作系统。

例如：

Java：

```
Java程序

↓

JVM

↓

操作系统
```

------

Go：

类似：

```
Go程序

↓

Go Runtime

↓

操作系统
```

------

Runtime 负责：

- Goroutine 调度
- 内存管理
- GC
- Channel
- Timer
- 网络事件

------

所以：

Go 程序不是直接运行。

中间：

有一个 Runtime。

------

# 第二节：为什么需要 Runtime？

假设：

没有 Runtime。

------

创建 Goroutine：

操作系统线程。

------

100万个任务：

需要：

100万个线程。

------

不可能。

------

所以：

Go 自己管理。

------

# 第三节：Goroutine 本质是什么？

很多人误解：

Goroutine = 小线程。

------

不准确。

------

实际上：

Goroutine 是：

Runtime 管理的任务结构。

------

内部：

类似：

```
type g struct{

    stack

    status

    program counter

}
```

------

保存：

执行状态。

------

# 第四节：G-M-P 模型

这是 Go 调度核心。

三个角色：

------

## G

Goroutine。

表示：

一个任务。

------

## M

Machine。

表示：

操作系统线程。

------

## P

Processor。

表示：

调度资源。

------

关系：

```
       P

       |

       M

       |

       G
```

------

# 第五节：G-M-P 详细理解

假设：

电脑：

8 核 CPU。

------

Go 创建：

10000 个 Goroutine。

------

不会：

创建 10000 线程。

------

而是：

```
G1
G2
G3
...
G10000


↓

Scheduler


↓

M1 M2 M3...


↓

CPU
```

------

------

# 第六节：P 的作用

P：

不是 CPU。

------

它表示：

执行 Go 代码所需资源。

------

包含：

- 本地 Goroutine 队列
- 调度信息

------

通常：

P 数量：

等于 CPU 核心数。

------

例如：

8 核：

默认：

8 个 P。

------

查看：

```
runtime.GOMAXPROCS()
```

------

# 第七节：Goroutine 调度过程

例如：

代码：

```
go task()
```

------

发生：

步骤：

------

第一步：

创建 G。

------

第二步：

放入队列。

------

第三步：

P 获取。

------

第四步：

M 执行。

------

过程：

```
创建 G

↓

等待调度

↓

P 获取

↓

M 执行

↓

完成
```

------

# 第八节：为什么 Goroutine 快？

因为：

切换成本低。

------

线程切换：

需要：

操作系统参与。

------

Goroutine：

Runtime 自己切换。

------

成本：

低很多。

------

# 第九节：Goroutine Stack

线程：

通常：

固定栈。

------

例如：

1MB。

------

Goroutine：

开始：

很小。

------

例如：

几 KB。

------

需要：

自动增长。

------

结构：

```
开始

小 Stack


↓

需要更多


↓

扩容
```

------

# 第十节：Stack 为什么重要？

因为：

函数调用：

需要栈。

例如：

```
func A(){

    B()

}


func B(){

}
```

------

保存：

- 参数
- 局部变量
- 返回地址

------

都在 Stack。

------

# 第十一节：抢占式调度

以前：

Go：

协作式调度。

------

问题：

如果：

一个 Goroutine：

死循环：

```
for{

}
```

------

可能：

占用 CPU。

------

现在：

Go 支持：

抢占。

------

Runtime：

可以暂停它。

------

# 第十二节：抢占过程

例如：

G1：

执行很久。

------

Runtime：

发现：

超过时间。

------

插入：

调度点。

------

切换：

G2。

------

类似：

操作系统线程调度。

------

# 第十三节：GC 是什么？

GC：

Garbage Collection。

垃圾回收。

------

作用：

自动释放：

不用的内存。

------

例如：

```
func test(){

    a:=make([]int,1000)

}
```

------

函数结束。

------

a：

没有引用。

------

GC：

回收。

------

# 第十四节：为什么需要 GC？

如果：

手动管理。

例如 C：

```
malloc()

free()
```

------

容易：

- 内存泄漏
- 野指针
- double free

------

Go：

自动管理。

------

# 第十五节：Go GC 基本流程

现代 Go：

三色标记清除。

------

对象：

分三个状态：

------

## 白色

未知。

------

## 灰色

发现。

等待扫描。

------

## 黑色

确认存活。

------

过程：

```
白

↓

灰

↓

黑
```

------

# 第十六节：GC 工作过程

简单理解：

1. ​

停止分配压力。

------

1. ​

扫描对象。

------

1. ​

标记存活。

------

1. ​

清理垃圾。

------

# 第十七节：为什么 GC 会影响性能？

因为：

GC 需要 CPU。

------

例如：

大量创建：

```
for{

    make([]byte,100000)

}
```

------

产生：

大量垃圾。

------

GC：

频繁运行。

------

性能下降。

------

# 第十八节：内存逃逸是什么？

这是 Go 性能重点。

------

看：

```
func create(){

    x:=10

}
```

------

x：

在哪里？

------

可能：

Stack。

------

但是：

```
func create()*int{

    x:=10

    return &x

}
```

------

x：

必须活下来。

------

所以：

放 Heap。

------

这叫：

逃逸。

------

# 第十九节：Stack 和 Heap 区别

|          | Stack        | Heap |
| -------- | ------------ | ---- |
| 速度     | 快           | 慢   |
| 管理     | 函数结束释放 | GC   |
| 生命周期 | 短           | 长   |

------

原则：

尽量：

Stack。

------

# 第二十节：如何查看逃逸？

命令：

```
go build -gcflags="-m"
```

------

例如：

输出：

```
moved to heap
```

------

表示：

逃逸。
