# 第七节：一个真实的小故事

据 Go 团队分享，Go 的最初设计工作，是从一个很朴素的问题开始的：

> **"如果今天重新设计一门系统编程语言，我们会保留什么，又会去掉什么？"**

他们没有先讨论要加入多少新特性。

相反，他们不断讨论：

- 哪些特性能让团队更高效？
- 哪些特性虽然强大，却容易被滥用？
- 哪些设计会让十年后的维护者更轻松？

这种"先做减法，再做加法"的思路，贯穿了 Go 的整个设计过程。

因此，当你以后觉得：

> "为什么 Go 没有 XXX？"

可以先换一个问题：

> **"Go 的设计者为什么认为，不加入 XXX 会更好？"**

很多答案都会变得清晰。

# 第八节：Go 的价值观，来自三位设计者

现在，我们把三个人的特点放在一起。

| 人物             | 带来的影响                            |
| ---------------- | ------------------------------------- |
| Ken Thompson     | Unix 哲学、简单工具、组合思想         |
| Rob Pike         | Go 的设计哲学、简洁、可读性、工程效率 |
| Robert Griesemer | 编译器、性能、工具链、语言实现        |

Go 不是某个人天马行空的想法。

而是三种经验融合后的结果。

# 本章总结

请记住下面五句话。

### 第一

Go 的设计不是偶然。

它来自几十年的工程经验。

------

### 第二

Go 很多设计，都能看到 Unix 思想的影子。

------

### 第三

Go 不追求"功能最多"。

而追求：

> **长期维护成本最低。**

------

### 第四

Go 的工具链如此统一。

不是限制程序员。

而是减少团队协作成本。

------

### 第五（本章最重要）

> **Go 的每一个"没有"，几乎都是一次经过深思熟虑的选择。**

以后学习任何 Go 特性，都请先记住这句话。

# 企业案例

假设你加入一家创业公司。

团队有 8 个人。

现在有两种方案：

方案 A：

- 每个人用自己喜欢的代码格式。
- 每个人用不同的构建工具。
- 每个人选择不同的项目结构。

方案 B：

- 所有人统一格式（`gofmt`）。
- 所有人统一构建（`go build`）。
- 所有人统一测试（`go test`）。
- 所有人遵循相同的项目组织方式。

短期来看，方案 A 看似更自由。

长期来看，你认为哪种方案维护成本更低？为什么？
