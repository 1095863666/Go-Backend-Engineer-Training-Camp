# Interface（二）——interface 底层结构：为什么 nil 接口会产生隐藏 Bug？

------

> **一句话总结本章**

> **Go 的 interface 不是一个简单指针，它内部保存了“类型信息 + 数据地址”，所以“类型为空”和“数据为空”是两回事。**

------

# 本章学习目标

完成本章后，你应该能够：

- 理解 interface 内部结构
- 理解 nil interface
- 理解 typed nil
- 解释为什么 interface != nil
- 理解空接口 `interface{}`
- 理解 `any`
- 避免企业项目中的接口陷阱

------

# 第一节：回顾 interface

上一章：

我们定义：

```
type Speaker interface {

    Speak()

}
```

然后：

```
type Dog struct{

}


func (Dog) Speak(){

}
```

Dog：

自动实现：

Speaker。

------

现在：

问题来了。

变量：

```
var s Speaker
```

这个变量：

里面是什么？

------

很多新人认为：

```
s = nil
```

结束。

但是：

interface 比普通变量复杂。

------

# 第二节：interface 本质是什么？

Go 官方没有直接暴露结构。

但是可以理解成：

```
type interface struct {

    typeInfo

    data

}
```

也就是：

两个部分。

------

## 第一部分：动态类型

例如：

```
Dog
```

表示：

当前保存什么类型。

------

## 第二部分：动态值

例如：

```
Dog{}
```

表示：

真正的数据。

------

所以：

interface：

类似：

```
+----------------+
| 类型信息       |
+----------------+
| 数据地址       |
+----------------+
```

------

# 第三节：普通 interface

看：

```
var s Speaker
```

现在：

没有赋值。

内部：

```
type:

nil


data:

nil
```

两个都是空。

所以：

```
s == nil
```

结果：

```
true
```

------

这种：

叫：

真正的 nil interface。

------

# 第四节：赋值之后发生什么？

例如：

```
dog := Dog{}

var s Speaker

s = dog
```

现在：

interface：

内部：

变成：

```
type:

Dog


data:

Dog{}
```

------

虽然：

Dog：

没有指针。

但是：

interface 已经知道：

它装的是：

Dog。

------

所以：

```
s == nil
```

结果：

```
false
```

------

为什么？

因为：

类型信息存在。

------

# 第五节：最经典 Bug —— typed nil

现在看：

```
type MyError struct{

}


func (e *MyError) Error() string{

    return "error"

}
```

MyError：

实现：

error 接口。

------

现在：

```
var e *MyError = nil

var err error = e
```

问题来了。

------

很多人认为：

err：

应该是：

nil。

但是：

实际上：

不是。

------

为什么？

因为：

interface：

内部：

现在：

```
type:

*MyError


data:

nil
```

------

注意：

类型存在。

------

所以：

```
err == nil
```

结果：

```
false
```

------

# 第六节：为什么这是危险的？

真实项目：

大量：

这样写：

```
func GetUser() error {

    var err *MyError

    return err

}
```

调用：

```
err := GetUser()


if err != nil {

    log.Println("失败")

}
```

------

程序：

进入：

错误分支。

但是：

实际上：

没有错误。

------

这就是：

隐藏 Bug。

------

# 第七节：正确写法

不要：

返回：

typed nil。

错误：

```
func Test() error {

    var e *MyError

    return e

}
```

------

正确：

```
func Test() error {

    return nil

}
```

------

或者：

真正创建：

错误：

```
return &MyError{}
```

------

原则：

> 返回 error 接口时，要返回 nil，而不是一个装着 nil 指针的接口。

------

# 第八节：空接口是什么？

以前：

Go：

写：

```
interface{}
```

表示：

任何类型。

例如：

```
var x interface{}

x = 100

x = "hello"

x = true
```

都可以。

------

为什么？

因为：

空接口：

没有要求：

任何方法。

------

任何类型：

都满足：

空接口。

------

# 第九节：any 是什么？

Go 1.18：

增加：

```
any
```

其实：

只是：

别名。

等价：

```
type any = interface{}
```

------

所以：

下面：

一样：

```
var a any

var b interface{}
```

------

企业：

现在：

更推荐：

使用：

```
any
```

因为：

语义：

更清晰。

------

# 第十节：为什么 interface{} 不应该乱用？

很多新人：

喜欢：

```
func Process(data interface{})
```

觉得：

万能。

------

但是：

问题：

类型丢失。

例如：

```
data := interface{}
```

里面：

到底：

是什么？

不知道。

------

你必须：

类型断言。

------

# 第十一节：类型断言

例如：

```
value,ok := data.(string)
```

意思：

尝试：

把 data：

转换成：

string。

------

如果：

成功：

```
value = 字符串

ok = true
```

失败：

```
ok = false
```

------

推荐：

使用：

安全写法。

------

不要：

```
value := data.(string)
```

因为：

失败：

直接 panic。

------

# 第十二节：类型 switch

例如：

```
switch v := data.(type){

case string:

    fmt.Println(v)


case int:

    fmt.Println(v)

}
```

------

常用于：

处理：

不同类型。

------

# 第十三节：企业什么时候使用 interface{}？

现在：

any。

常见：

场景：

------

## JSON

例如：

```
map[string]any
```

因为：

JSON：

值类型：

不固定。

------

## 配置

例如：

动态配置。

------

## 通用工具

例如：

序列化。

------

但是：

业务核心：

尽量避免。

------

# 第十四节：Flow Engine 实战

我们的流程变量：

之前：

设计：

```
Variables map[string]any
```

为什么？

因为：

流程参数：

可能：

不同类型。

例如：

```
{
 "temperature":30,
 "name":"test",
 "enabled":true
}
```

------

所以：

这里：

使用：

any。

合理。

------

但是：

节点执行：

不要：

这样：

```
func Execute(data any)
```

因为：

节点不知道：

输入是什么。

------

更好：

定义：

```
type NodeContext struct {

    Variables map[string]any

}
```

------

接口：

负责：

行为。

结构体：

负责：

数据。

------

# 第十五节：接口设计原则（高级）

现在：

总结：

接口：

有三个层次。

------

## 第一层

能力抽象。

例如：

```
Reader
Writer
```

------

## 第二层

依赖隔离。

例如：

业务依赖：

Repository。

------

## 第三层

运行时多态。

例如：

不同 NodeExecutor。

------

# 第十六节：接口最佳实践

记住：

------

## 1.

不要返回大接口。

例如：

不好：

```
func GetUser() interface{}
```

------

好：

```
func GetUser() (*User,error)
```

------

## 2.

接口应该小。

------

## 3.

接口应该由使用方定义。

------

## 4.

不要为了接口而接口。

------
