# Interface（一）——为什么 Go 接口不需要 implements？

------

> **一句话总结本章**

> **Go 接口描述的是“能力”，而不是“身份”；只要一个类型拥有接口要求的方法，它自动满足接口。**

------

# 本章学习目标

完成本章后，你应该能够：

- 理解什么是接口
- 理解 Go 接口设计思想
- 理解隐式实现
- 理解鸭子类型
- 理解接口为什么降低耦合
- 能设计简单业务接口

------

# 第一节：为什么需要接口？

先看一个问题。

假设：

我们开发支付系统。

现在支持：

支付宝。

```
func PayByAlipay(){

}
```

微信：

```
func PayByWechat(){

}
```

银行卡：

```
func PayByBank(){

}
```

------

如果业务代码：

这样：

```
if payType=="alipay"{

}

if payType=="wechat"{

}
```

会发生什么？

------

以后：

增加：

PayPal。

Apple Pay。

Crypto。

代码：

越来越乱。

------

我们需要：

一种方式：

告诉程序：

> 不管你是什么支付方式，只要你能支付，就可以使用。

------

这就是：

接口。

------

# 第二节：接口是什么？

接口：

定义：

一组行为。

例如：

支付：

需要：

支付方法。

```
type Payment interface {

    Pay(amount float64) error

}
```

意思：

任何类型。

只要有：

```
Pay(float64) error
```

这个方法。

就是：

Payment。

------

注意：

接口：

不是数据。

是：

能力描述。

------

# 第三节：传统接口思想

Java：

例如：

```
interface Payment {

    void Pay();

}
```

实现：

```
class Alipay implements Payment{

}
```

这里：

程序员主动声明：

我实现它。

------

这是：

显式接口。

------

# 第四节：Go 接口思想

Go：

没有：

implements。

例如：

```
type Alipay struct{

}


func (a Alipay) Pay(amount float64) error{

    return nil

}
```

然后：

直接：

使用：

```
var p Payment

p = Alipay{}
```

成功。

------

为什么？

因为：

Alipay：

拥有：

Pay 方法。

------

它自动满足：

Payment。

------

# 第五节：什么叫隐式实现？

一句话：

> **类型不需要声明自己属于接口，编译器自动判断。**

------

例如：

接口：

```
type Animal interface{

    Speak()

}
```

类型：

```
type Dog struct{}

func (Dog) Speak(){

}
```

------

Dog：

没有：

写：

```
implements Animal
```

但是：

它就是：

Animal。

------

因为：

它具备：

这个能力。

------

# 第六节：为什么 Go 要这样设计？

核心：

降低耦合。

------

假设：

你写一个日志系统。

传统：

```
class Logger implements Storage
```

Storage：

提前定义。

------

问题：

第三方库：

怎么办？

------

Go：

不需要。

只要：

满足方法。

直接使用。

------

例如：

你的代码：

```
func Save(s Storage){

}
```

------

别人：

写：

```
type MyStorage struct{

}

func (m MyStorage) Save(){

}
```

马上：

可以接入。

------

不用：

修改原代码。

------

# 第七节：鸭子类型（Duck Typing）

这是接口思想来源。

英文：

Duck Typing。

------

一句话：

> 如果它走起来像鸭子，叫起来像鸭子，那么它就是鸭子。

------

例如：

接口：

```
type Duck interface{

    Quack()

}
```

------

类型：

```
type Robot struct{}

func(Robot) Quack(){

}
```

------

Robot：

不是鸭子。

但是：

它有：

Quack。

所以：

程序：

认为：

它具备鸭子的能力。

------

Go：

大量使用：

这种思想。

------

# 第八节：接口和具体类型区别

例如：

```
type User struct{

    Name string

}
```

User：

描述：

数据。

------

接口：

```
type Saver interface{

    Save()

}
```

Saver：

描述：

行为。

------

所以：

记住：

结构体：

回答：

> 它是什么？

接口：

回答：

> 它能做什么？

------

# 第九节：Flow Engine 实例

我们的流程系统。

有很多执行节点：

例如：

SQL 节点。

HTTP 节点。

脚本节点。

消息节点。

------

如果：

不用接口：

可能：

这样：

```
func ExecuteSQLNode(){

}

func ExecuteHTTPNode(){

}

func ExecuteScriptNode(){

}
```

------

以后：

100 种节点。

怎么办？

------

使用接口：

```
type NodeExecutor interface {

    Execute(ctx Context) error

}
```

------

SQL：

```
type SQLNode struct{

}


func (s SQLNode) Execute(ctx Context) error{

    return nil

}
```

------

HTTP：

```
type HTTPNode struct{

}


func (h HTTPNode) Execute(ctx Context) error{

    return nil

}
```

------

然后：

统一：

```
func RunNode(
    executor NodeExecutor,
) error{

    return executor.Execute(ctx)

}
```

------

新增节点：

不用修改：

RunNode。

------

这就是：

扩展性。

------

# 第十节：接口减少依赖

看数据库。

不好：

```
func CreateUser(
    db *mysql.DB,
)
```

问题：

业务代码：

绑定：

MySQL。

------

以后：

换 PostgreSQL。

怎么办？

------

更好：

```
type UserRepository interface{

    Save(user User) error

}
```

------

MySQL：

实现。

PostgreSQL：

实现。

------

业务：

只依赖：

接口。

------

这就是：

企业架构里面：

非常重要的：

依赖倒置。

------

# 第十一节：Go 接口的小接口思想

Go 非常喜欢：

小接口。

例如：

不要：

```
type UserManager interface{

    Create()

    Update()

    Delete()

    Query()

    Export()

}
```

太大。

------

更好：

```
type Creator interface{

    Create()

}
```

------

为什么？

因为：

小接口：

容易实现。

容易测试。

容易组合。

------

# 第十二节：标准库中的接口

Go 标准库：

大量使用：

接口。

例如：

------

Reader：

```
type Reader interface{

    Read([]byte)(int,error)

}
```

------

Writer：

```
type Writer interface{

    Write([]byte)(int,error)

}
```

------

文件：

实现。

网络：

实现。

Buffer：

实现。

------

所以：

任何地方：

都可以：

接入。

------

# 第十三节：接口的陷阱

接口：

非常强。

但是：

不要滥用。

------

错误：

为了接口而接口。

例如：

```
type UserService interface{

}
```

只有：

一个实现。

没有：

变化可能。

没有意义。

------

原则：

> 有变化方向时，再抽象接口。

------

# 第十四节：接口设计原则

企业经验：

记住：

------

## 原则 1

接口属于使用方。

不是实现方。

------

什么意思？

例如：

业务需要：

保存。

业务定义：

```
type UserSaver interface{

 Save()

}
```

------

不是：

数据库模块：

主动定义。

------

## 原则 2

接口越小越好。

------

## 原则 3

不要提前抽象。

------
