# Package 包管理——Go 为什么强调组合，而不是复杂类层级？

------

> **一句话总结本章**

> **Go 使用 package 作为代码组织单位，通过清晰的边界和组合方式构建大型系统，而不是依赖复杂的继承体系。**

------

# 本章学习目标

完成本章后，你应该能够：

- 理解 package 是什么
- 理解 package 和目录关系
- 理解 module
- 理解 import 工作方式
- 理解导出规则
- 理解 internal 包
- 能设计企业级 Go 项目结构

------

# 第一节：为什么需要 package？

假设：

现在写一个商城。

最开始：

```
main.go
```

里面：

```
用户代码

订单代码

支付代码

数据库代码
```

------

1000 行以后：

问题：

出现。

------

## 问题 1

不知道代码属于哪里。

------

## 问题 2

修改一个地方：

影响全部。

------

## 问题 3

无法复用。

------

所以：

需要：

模块化。

------

Go：

使用：

package。

------

# 第二节：package 是什么？

简单理解：

> package 是一组相关 Go 文件的集合。

例如：

目录：

```
user
|
|-- user.go
|-- service.go
|-- repository.go
```

里面：

都有：

```
package user
```

------

那么：

它们属于：

同一个 package。

------

# 第三节：package 和目录关系

Go 有一个重要规则：

> 一个目录通常对应一个 package。

例如：

```
project

├── user

│   ├── user.go

│   └── service.go


├── order

│   ├── order.go
```

------

对应：

```
package user
```

和：

```
package order
```

------

------

# 第四节：package 名称规则

例如：

目录：

```
flow/node
```

文件：

```
package node
```

------

通常：

目录名：

和：

package 名：

一致。

------

但是：

不是强制。

例如：

目录：

```
utils
```

里面：

可以：

```
package helper
```

但是：

不推荐。

------

企业习惯：

保持一致。

------

# 第五节：main package

Go 程序入口：

特殊。

必须：

```
package main
```

------

并且：

必须有：

```
func main(){

}
```

------

例如：

```
package main


func main(){

    startServer()

}
```

------

编译：

生成：

可执行文件。

------

# 第六节：import 是什么？

例如：

```
import "fmt"
```

意思：

引入：

fmt package。

------

然后：

使用：

```
fmt.Println()
```

------

package：

提供：

公开能力。

------

# 第七节：Go 如何判断公开？

Go：

没有：

public/private。

而是：

大小写。

------

## 大写开头

公开。

例如：

```
type User struct{

}
```

其他 package 可以使用。

------

## 小写开头

私有。

例如：

```
type user struct{

}
```

只能：

当前 package 使用。

------

这是 Go 很独特的设计。

------

# 第八节：为什么不用 public/private？

因为：

Go 追求：

简单。

------

Java：

```
public class User

private String name

protected
```

规则很多。

------

Go：

一个规则：

> 首字母大小写决定可见性。

------

# 第九节：package 设计思想

重要。

不要：

按照技术分类。

例如：

不好：

```
controllers

services

models

utils
```

这种：

Java 风格。

------

Go 更推荐：

按照业务领域。

例如：

```
user

order

payment

workflow
```

------

为什么？

因为：

业务变化：

通常：

一起变化。

------

# 第十节：Flow Engine 项目结构

例如：

设计：

流程引擎。

推荐：

```
flow-engine

│       └── main.go

├── pkg


│   └── logger


├── api


│   └── http


└── go.mod
```

------

# 第十一节：cmd 目录

企业项目：

常见：

```
cmd
```

表示：

程序入口。

------

例如：

多个服务：

```
cmd

├── api

├── worker

└── scheduler
```

------

每个：

一个 main。

------

# 第十二节：internal 包

这是 Go 非常重要的设计。

------

例如：

```
internal/payment
```

里面：

代码。

------

规则：

外部项目：

不能 import。

------

例如：

你的项目：

```
github.com/company/project/internal/user
```

------

别人：

不能：

```
import github.com/company/project/internal/user
```

------

为什么？

保护：

内部实现。

------

# 第十三节：为什么需要 internal？

大型项目：

一定有：

内部代码。

------

例如：

你的：

数据库实现。

不希望：

别人依赖。

------

否则：

以后：

改数据库。

别人项目：

全部炸。

------

internal：

帮助：

控制依赖。

------

# 第十四节：pkg 目录

常见：

```
pkg
```

表示：

可以被外部使用的公共代码。

例如：

```
pkg/logger

pkg/cache

pkg/validator
```

------

但是：

不是强制。

只是：

社区习惯。

------

# 第十五节：module 是什么？

package：

解决：

代码组织。

------

module：

解决：

项目管理。

------

例如：

创建：

```
go mod init github.com/company/project
```

生成：

```
go.mod
```

------

里面：

记录：

项目名称。

依赖版本。

------

# 第十六节：go.mod

例如：

```
module github.com/example/app


go 1.22


require (

    gin v1.9.0

)
```

------

表示：

当前项目：

需要：

Gin。

------

# 第十七节：package 和 module 区别

很多新人混淆。

------

package：

代码模块。

例如：

```
user
```

------

module：

整个项目。

例如：

```
github.com/company/app
```

------

关系：

```
module

|

|-- package

|-- package

|-- package
```

------

# 第十八节：循环依赖问题

Go：

禁止：

package 循环引用。

例如：

```
user

↓

order


order

↓

user
```

------

为什么？

因为：

依赖关系混乱。

------

Go 强制：

让架构清晰。

------

# 第十九节：如何解决循环依赖？

通常：

抽象接口。

例如：

不要：

user 直接调用 order。

定义：

interface。

------

或者：

抽公共部分：

```
common
```

------

但是：

不要：

随便建：

utils。

------

# 第二十节：Flow Engine 架构实践

例如：

Node 执行。

不要：

flow package：

直接依赖：

所有节点。

------

更好：

定义：

interface：

```
type NodeExecutor interface{

    Execute() error

}
```

------

节点：

自己注册。

------

结构：

```
flow

|

interface


node/sql

|

implementation
```

------

这样：

新增节点：

不用修改核心。

------

# 第二十一节：package 设计原则

记住：

------

## 原则 1

应该表达职责。

好：

巨大 package。

例如：

```

```

所有业务。

------

## 原则 3

package：

应该：

高内聚。
