# 函数（Function）——为什么 Go 选择多返回值，而不是异常？

> **一句话总结本章**

# 第一节：为什么需要函数？

先看：

没有函数。

例如：

用户注册：

```
// 验证用户名

// 检查密码

// 保存数据库

// 发送邮件
```

如果：

注册逻辑：

写 10 次。

怎么办？

复制？

------

复制的问题：

- 修改困难
- 容易遗漏
- 无法测试

------

所以：

把重复逻辑：

抽出来。

形成：

函数。

------

# 第二节：函数本质是什么？

简单理解：

函数：

就是：

一段：

有名字的代码。

例如：

```
func Add(a int,b int) int {

    return a+b

}
```

它表示：

输入：

两个整数。

输出：

一个整数。

------

数学：

类似：

```
f(x,y)=x+y
```

------

但是：

工程里面：

函数更重要的作用：

是：

隐藏复杂性。

------

例如：

你调用：

```
SaveUser(user)
```

你不需要知道：

内部：

- SQL
- 网络
- 事务

这些：

都被函数隐藏。

------

# 第三节：Go 函数基本结构

标准：

```
func 函数名(参数) 返回类型 {

    函数体

}
```

例如：

```
func Add(a int,b int) int {

    return a+b

}
```

------

拆开：

```
func

↓

声明函数


Add

↓

函数名字


(a int,b int)

↓

输入


int

↓

输出
```

------

# 第四节：参数类型简写

例如：

```
func Add(a int,b int)
```

可以：

写成：

```
func Add(a,b int)
```

因为：

连续参数：

类型一样。

------

但是：

下面：

不可以：

```
func Test(a,b int,c string)
```

需要：

保持。

------

# 第五节：Go 参数传递

非常重要。

Go：

只有一种：

传递方式：

> 值传递。

------

例如：

```
func Change(x int){

    x=100

}


func main(){

    a:=10

    Change(a)

    fmt.Println(a)

}
```

输出：

```
10
```

------

因为：

复制。

------

# 第六节：那么指针怎么回事？

上一章：

我们讲：

指针。

例如：

```
func Change(x *int){

    *x=100

}
```

调用：

```
Change(&a)
```

------

注意：

Go 还是：

值传递。

只是：

复制的是：

地址。

------

进入函数：

得到：

地址副本。

两个指针：

指向：

同一个数据。

------

# 第七节：为什么 Go 支持多返回值？

这是 Go 最有特色的设计之一。

例如：

很多语言：

返回：

一个对象。

表示：

多个结果。

------

Go：

直接：

返回多个。

例如：

```
func Divide(a,b int)(int,error){

}
```

返回：

```
结果

错误
```

------

调用：

```
result,err:=Divide(10,2)
```

------

简单。

直接。

------

# 第八节：为什么不用 tuple？

Python：

```
return value,error
```

返回：

tuple。

------

Go：

没有。

为什么？

因为：

Go 希望：

返回值：

明确。

------

例如：

```
user,err:=GetUser()
```

一眼：

知道：

两个东西：

是什么。

------

# 第九节：错误处理为什么返回 error？

这是 Go 最大特色。

------

很多语言：

例如：

Java：

```
try{

}
catch(Exception e){

}
```

------

Go：

不用：

异常。

而是：

```
user,err:=GetUser()

if err != nil {

}
```

------

为什么？

因为：

错误：

也是：

程序正常流程的一部分。

------

# 第十节：为什么异常不是 Go 风格？

假设：

一个函数：

可能：

失败。

异常：

隐藏：

失败路径。

例如：

```
User getUser()
```

你不知道：

会不会失败。

------

Go：

直接：

告诉你：

```
func GetUser()(User,error)
```

------

函数签名：

就是：

契约。

------

# 第十一节：error 是什么？

注意。

error：

不是：

关键字。

它是：

接口。

以后：

详细讲接口。

现在：

理解：

它就是：

一种普通值。

------

例如：

```
type error interface {

    Error() string

}
```

------

所以：

错误：

可以：

传递。

保存。

包装。

------

# 第十二节：错误处理标准写法

企业 Go：

大量：

这样：

```
user,err:=GetUser(id)

if err != nil {

    return err

}

process(user)
```

------

不要：

这样：

```
if err==nil{

}
```

虽然：

可以。

但是：

Go 风格：

更喜欢：

提前返回。

------

# 第十三节：函数返回设计

不好：

```
func CreateUser(
    string,
    string,
    int,
    string,
)
```

问题：

参数：

太多。

调用：

不知道：

什么意思。

------

例如：

```
CreateUser(
    "Tom",
    "123456",
    18,
    "xxx",
)
```

谁知道：

第四个：

是什么？

------

更好：

使用结构体：

```
type CreateUserRequest struct{

    Name string

    Password string

    Age int

}
```

然后：

```
func CreateUser(req CreateUserRequest)
```

------

# 第十四节：函数应该多大？

Go 推崇：

小函数。

不好：

```
func ProcessOrder(){

    //500 行

}
```

------

更好：

拆：

```
ProcessOrder

    |

    ValidateOrder

    |

    CalculatePrice

    |

    SaveOrder

    |

    SendMessage
```

------

为什么？

因为：

每个函数：

只有：

一个责任。

------

# 第十五节：Flow Engine 实战

我们的流程执行器：

例如：

启动节点：

不要：

写：

```
func Start(
    int,
    string,
    bool,
    map[string]interface{},
)
```

------

应该：

```
type StartNodeRequest struct {

    NodeID NodeID

    Operator UserID

    Params map[string]any

}


func StartNode(
    req StartNodeRequest,
)(*Node,error)
```

------

返回：

节点。

错误。

清晰。

------

现在：

就是：

函数。

------

用途：

很多。

后面：

Gin 源码：

大量使用。


括号：

立即执行。

------

goroutine。

# 第十八节：函数设计原则

企业经验：

记住：

------

## 原则 1

函数名字：

表达行为。

好：

```
CreateUser()
```

差：

```
DoSomething()
```

------

## 原则 2

输入输出明确。

好：

```
(User,error)
```

差：

隐藏状态。

------

## 原则 3

错误：

不要吞掉。

差：

```
if err!=nil{

    return nil

}
```

------

好：

```
return nil,err
```
