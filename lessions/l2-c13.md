# Struct 结构体——为什么 Go 没有 Class，却能构建大型系统？

------

> **一句话总结本章**

> **Go 不通过继承构建复杂系统，而是通过 struct + method + interface + composition 组合出强大的工程模型。**

------

# 第一节：为什么需要 struct？

之前：

我们学习变量。

例如：

```
age := 18
name := "Tom"
```

问题：

业务对象通常：

不是一个数据。

------

例如：

用户：

```
User

姓名

年龄

邮箱

状态
```

如果：

这样：

```
name string

age int

email string
```

代码：

很快混乱。

------

我们需要：

把相关数据：

组合起来。

------

这就是：

struct。

------

# 第二节：什么是 struct？

例如：

```
type User struct {

    Name string

    Age int

    Email string

}
```

定义：

一个：

User 类型。

------

创建：

```
user := User{

    Name:"Tom",

    Age:18,

}
```

------

内存：

类似：

```
User

+-------------+
| Name        |
+-------------+
| Age         |
+-------------+
| Email       |
+-------------+
```

------

# 第三节：struct 和 class 区别

很多 Java 开发者：

第一反应：

struct = class。

但是：

不是。

------

Java:

```
class User {

    String name;

    void Login(){

    }

}
```

数据：

和：

行为：

绑定。

------

Go:

数据：

```
type User struct{

    Name string

}
```

行为：

单独定义：

```
func (u User) Login(){

}
```

------

Go：

故意分开。

------

# 第四节：为什么 Go 不要 Class？

因为：

Go 认为：

传统 OOP 有一些问题。

------

## 问题 1：继承容易复杂

例如：

Java：

```
Animal

  |

 Mammal

  |

 Dog
```

最后：

继承层次：

越来越深。

------

Go：

更喜欢：

组合。

------

## 问题 2：行为和数据绑定过紧

Go：

希望：

数据：

独立。

行为：

通过：

接口组合。

------

# 第五节：struct 方法

例如：

```
type User struct{

    Name string

}
```

添加方法：

```
func (u User) SayHello(){

    fmt.Println(u.Name)

}
```

------

调用：

```
user.SayHello()
```

------

这里：

`u`

叫：

receiver。

------

# 第六节：receiver 是什么？

receiver：

表示：

这个方法属于谁。

例如：

```
func (u User) Login()
```

意思：

User 有一个 Login 方法。

------

类似：

其他语言：

```
user.login()
```

------

但是：

Go：

没有隐藏：

this。

------

# 第七节：值 receiver

例如：

```
func (u User) Update(){

}
```

这里：

传入：

User 副本。

------

例如：

```
func (u User){

    u.Name="Jack"

}
```

外部：

不会改变。

------

原因：

值复制。

------

# 第八节：指针 receiver

例如：

```
func (u *User) Update(){

    u.Name="Jack"

}
```

------

调用：

```
user.Update()
```

Go：

自动：

取地址。

等价：

```
(&user).Update()
```

------

现在：

修改：

原对象。

------

# 第九节：什么时候使用指针 receiver？

企业：

非常常见。

------

## 情况 1

需要修改对象。

例如：

```
func (u *User) SetName()
```

------

## 情况 2

struct 很大。

避免复制。

例如：

```
type Context struct{

    //大量字段

}
```

------

## 情况 3

保持一致性。

例如：

一个类型：

大部分方法：

都是：

指针 receiver。

------

# 第十节：值 receiver 的场景

适合：

小对象。

不可变对象。

例如：

```
type Money struct{

    Amount int

}
```

------

方法：

只读取。

```
func (m Money) String() string
```

------

# 第十一节：struct 可以嵌套

例如：

用户：

有地址。

```
type Address struct{

    City string

}


type User struct{

    Name string

    Address Address

}
```

------

使用：

```
user.Address.City
```

------

# 第十二节：组合 Composition

这是 Go 替代继承的核心。

------

例如：

传统：

```
Animal

  |

Dog
```

------

Go：

```
type Animal struct{

    Name string

}


type Dog struct{

    Animal

}
```

------

Dog：

拥有：

Animal。

------

这叫：

嵌入。

------

# 第十三节：方法提升

例如：

```
func (a Animal) Eat(){

}
```

------

Dog：

自动拥有：

```
dog.Eat()
```

------

但是：

注意：

这不是继承。

------

只是：

组合。

------

# 第十四节：为什么组合更好？

假设：

动物：

会吃。

会跑。

会飞。

------

继承：

容易爆炸。

------

组合：

自由组合：

```
Dog

+
Eat

+
Run
```

------

Bird:

```
Bird

+
Eat

+
Fly
```

------

更加灵活。

------

# 第十五节：Flow Engine 领域建模

------

## Flow

```
}
```

------

## Node

```

    Status NodeStatus

```

------

## Execution

```
type Execution struct {

}
```

------

这就是：

领域模型。

------

# 第十六节：struct + interface 配合

企业最常见模式：

------

数据：

```
type User struct{

}
```

------

能力：

```
type Repository interface{

    Save(User) error

}
```

------

实现：

```
type MysqlRepository struct{

}
```

------

组合：

形成完整系统。

------

# 第十七节：Go 面向对象总结

Go 不是没有 OOP。

而是：

换了一种方式。

------

传统 OOP：

```
Class

↓

Inheritance

↓

Override
```

------

Go：

```
Struct

↓

Composition

↓

Interface
```

------

# 第十八节：常见错误

## 错误 1

把 struct 当 class。

写：

巨大 struct。

包含：

所有业务。

------

## 错误 2

receiver 混乱。

例如：

同一个类型：

一半：

值 receiver。

一半：

指针 receiver。

------

## 错误 3

过度嵌入。

不是所有关系：

都适合组合。
