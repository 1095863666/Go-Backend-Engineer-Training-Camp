# 第一节：时间回到 2007 年

现在，请你把自己想象成 Google 的一名工程师。

不是今天的 Google。

而是 **2007 年的 Google**。

那一年：

Google 已经拥有：

- Search（搜索）
- Gmail（邮箱）
- Maps（地图）
- Ads（广告）
- YouTube（2006 年收购）
- Android（即将发布）

全球每天都有海量请求进入 Google 的服务器。

对于普通用户来说，只是输入了一次搜索：

```
golang tutorial
```

但在 Google 内部，真正发生的事情更像这样：

```
                用户请求
                    │
                    ▼
             Google Frontend
                    │
        ┌───────────┼───────────┐
        ▼           ▼           ▼
    用户服务      搜索服务     广告服务
        │           │           │
        ▼           ▼           ▼
     缓存系统      索引系统     推荐系统
        │           │           │
        └───────────┼───────────┘
                    ▼
                 数据中心
```

一次搜索。

可能涉及：

几十个服务。

甚至上百次 RPC 调用。

很多人觉得：

程序就是：

```
main()

↓

完成
```

实际上。

大型互联网公司根本不是这样。

一个真实系统。

更像这样。

```
Search

├── frontend

├── auth

├── cache

├── ranking

├── log

├── monitor

├── storage

├── rpc

├── scheduler

├── metrics

└── ...
```

每一个目录。

都可能：

几十万行代码。

整个项目。

可能：

几百万行。

更重要的是：

**不是一个人在维护。**

而是：

```
500

↓

1000

↓

2000

↓

5000

工程师
```

共同开发。

# 第三节：真正的问题不是语言

请记住一句话。

> **Google 最大的问题，不是程序跑得慢，而是团队开发越来越慢。**

这是 Go 设计的核心。

很多初学者都会误以为：

> Go 是为了提高程序运行速度。

其实不是。

Google 最担心的是：

```
程序越来越大

↓

工程师越来越多

↓

代码越来越复杂

↓

维护越来越困难

↓

开发越来越慢
```

注意。

这里没有：

CPU。

没有：

内存。

没有：

性能。

而是：

**团队效率。**

------

# 第四节：为什么 C++ 开始变得痛苦？

这里先说明。

**C++ 是一门非常优秀的语言。**

今天：

Chrome

Unreal Engine

MySQL

LLVM

大量系统。

仍然使用 C++。

问题不是：

C++ 不好。

而是：

Google 的规模。

已经超过了 C++ 最舒服的范围。

------

一个新人。

继续。

又看到：

```

...

```

继续。

还有：

```
virtual
```

```
override
```

```
friend
```

```
template
```

```

```

```

```

……

请注意。

这里没有任何一种技术是错误的。

但是：

对于一个：

几千名工程师。

持续开发十年的项目。

意味着：

**新人需要很长时间才能真正理解项目。**

Google 开始思考：

> 有没有一种语言，可以让新人更快上手？

# 第五节：Google 不喜欢"聪明代码"

Rob Pike 曾表达过一个观点：

> 聪明的代码，通常也是难维护的代码。

来看两个例子。

第一种：

```
if ((a & 1) == 0 && (b << 2) > c) {
    ...
}
```

一个经验丰富的程序员。

可能觉得：

很帅。

但是：

半年以后。

他自己回来。

也可能要重新理解。

------

第二种。

```
if isEven(a) && isValid(b, c) {
    ...
}
```

Go 更喜欢第二种。

因为：

代码首先是给人看的。

其次。

才是机器执行。

这一点。

以后我们会不断提到。

------

# 第六节：Google 希望代码可以被"普通工程师"维护

这是 Go 一个非常重要的思想。

很多语言。

追求：

> 可以写出非常复杂、非常巧妙的代码。

Go 不追求。

Go 希望：

一个刚加入团队三个月的工程师。

也能够：

读懂代码。

修改代码。

提交代码。

这就是：

为什么 Go：

宁愿：

少一点语法。

少一点魔法。

少一点技巧。

------

# 第七节：编译速度越来越慢

这是很多人不知道的一件事。

假设：

Google 有：

```
200 万行 C++ 代码
```

修改：

一行。

重新编译。

可能需要：

```
几十分钟
```

甚至：

更久。

请想象一下。

一天：

你：

修改：

20 次。

每一次。

等待：

10 分钟。

意味着：

一天。

有几个小时。

都在：

等待编译。

Google 无法接受。

于是。

Go 提出了一个目标。

> **编译速度必须足够快。**

后来。

Go 的编译速度。

成为：

它最著名的特点之一。

------

# 第八节：Google 真正想要什么？

到这里。

你会发现。

Google 从来没有说：

我要最快的语言。

也没有说：

我要功能最多的语言。

他们真正想要的是：

```
简单

↓

容易读

↓

容易维护

↓

容易部署

↓

容易协作

↓

编译快

↓

天然支持并发
```

如果只能记一句。

请记住：

> **Go 是一门为了软件工程而设计的语言。**

不是：

为了语法设计。

不是：

为了性能设计。

而是：

为了：

**工程。**

# 本章总结

这一章，请你记住下面四句话。

------

### 第一

Google 最大的问题。

不是：

程序太慢。

而是：

**团队越来越慢。**

------

### 第二

Go 想提高的。

不是：

程序员一个人的效率。

而是：

整个团队。

几十人。

几百人。

几千人。

共同开发时的效率。

------

### 第三

Go 不喜欢：

过度设计。

过度抽象。

复杂技巧。

因为：

这些都会增加维护成本。

------

### 第四（本章最重要的一句话）

> **软件真正的成本，不是写代码，而是维护代码。**

请把这句话记下来。

以后。

Interface。

Context。

Package。

Error。

全部都会回到这句话。

## 作者的话

到这里，我们还没有写一行 Go。

这是**故意**的。

因为在我看来，**学语言之前，先理解它的价值观，比记住语法更重要。**

下一章（**Lesson 1 · Chapter 3**），我们将认识 Go 的三位设计者——**Rob Pike、Ken Thompson 和 Robert Griesemer**。

理解他们各自的背景后，你会发现：

- 为什么 Go 像 Unix 一样崇尚简单。
- 为什么 Go 的工具链如此统一。
- 为什么 Go 的很多设计看起来"保守"，却非常适合长期维护的大型工程。

这一章会是整门课程第一次真正接触 Go 的设计思想，也是后续所有内容的思想基础。
