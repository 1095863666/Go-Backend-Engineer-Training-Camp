# Go 的核心设计哲学：为什么 Go 故意舍弃很多功能？

------

> **一句话总结本章**

> **Go 的设计不是追求"拥有最多能力"，而是主动删除那些会增加长期复杂度的能力。**

------

# 本章学习目标

完成本章以后，你应该能够理解：

- 为什么 Go 没有传统意义上的 Class？
- 为什么 Go 没有继承？
- 为什么 Go 没有 try/catch？
- 为什么 Go 强调组合？
- 为什么 Go 喜欢显式，而不是隐藏行为？
- 为什么 Go 的"简单"是一种高级设计？

------

# 第一节：一个重要的问题

在学习 Go 的过程中。

你一定会遇到很多类似的问题。

例如：

> 为什么 Go 没有 class？

------

> 为什么 Go 没有继承？

------

> 为什么 Go 的错误处理这么麻烦？

------

> 为什么 Go 不像 Java 一样有异常？

------

> 为什么 Go 不提供更多语法糖？

------

很多初学者第一反应：

> Go 是不是功能不够？

这是一个非常自然的想法。

但是：

答案恰恰相反。

------

# 第二节：语言设计不是功能竞赛

很多人理解编程语言时，有一个误区：

认为：

```
语言能力 =
功能数量
```

例如：

A 语言：

100 个关键字。

B 语言：

50 个关键字。

于是认为：

A 更强。

------

但是实际工程不是这样。

一个语言增加一个功能：

看起来：

只是增加一点能力。

但是：

它同时增加：

- 学习成本
- 阅读成本
- 维护成本
- 沟通成本
- 设计选择

------

举一个简单例子。

假设一个语言支持：

10 种写循环方式。

理论上：

很强。

但是团队里面：

可能出现：

程序员 A：

```
for
```

程序员 B：

```
while
```

程序员 C：

函数式 map

程序员 D：

递归

------

最后：

一个简单循环。

出现：

4 种表达。

新人加入：

需要学习：

"为什么这里这样写？"

------

Go 的思想：

减少选择。

------

# 第三节：Go 为什么没有 Class？

这是很多 Java 程序员第一次学习 Go 最大的不适。

Java：

```
public class User {

    private String name;

    public void Login(){

    }

}
```

------

Go：

```
type User struct {

    Name string

}


func (u User) Login(){

}
```

------

看起来：

只是语法不同。

实际上：

思想完全不同。

------

Java 的核心：

对象。

```
Object

↓

Data + Behavior
```

数据：

放对象里面。

行为：

放对象里面。

------

Go：

更强调：

数据和行为可以分离。

```
Data

↓

Struct


Behavior

↓

Function
```

------

为什么？

因为大型系统中：

对象层级容易膨胀。

------

# 第四节：面向对象继承的问题

传统 OOP：

喜欢：

"is-a"

关系。

例如：

```
Animal

   ↑

 Dog
```

意思：

Dog 是一种 Animal。

------

小项目：

很好。

------

但是大型项目：

容易出现：

继承爆炸。

------

例如：

你设计一个游戏。

开始：

```
Animal

↓

Dog
```

后来需求增加。

需要：

会飞的狗。

会游泳的狗。

会攻击的狗。

于是：

```
Animal

├── Dog

├── FlyingDog

├── SwimmingDog

├── FightingDog

└── FlyingSwimmingFightingDog
```

慢慢：

类数量爆炸。

------

这就是：

经典问题：

## Class Explosion

类爆炸。

------

# 第五节：Go 为什么选择组合？

Go 更喜欢：

"has-a"

拥有关系。

------

例如：

不要：

```
FlyingDog

继承

Dog
```

而是：

```
Dog

+

FlyAbility
```

------

代码：

```
type FlyAbility struct {

}


type Dog struct {

    FlyAbility

}
```

------

意思：

Dog：

拥有飞行能力。

------

如果以后：

猫也会飞：

```
type Cat struct {

    FlyAbility

}
```

不用复制。

------

这就是：

组合。

------

# 第六节：组合为什么适合大型系统？

因为：

组合关系：

更松散。

------

继承：

像这样：

```
父类

  ↓

子类
```

修改父类。

可能影响：

所有子类。

------

组合：

像这样：

```
模块 A

模块 B

模块 C

组合使用
```

修改一个模块。

影响范围：

更小。

------

大型系统最怕：

修改一个地方。

引发：

连锁反应。

------

所以 Go 喜欢：

小模块。

自由组合。

------

# 第七节：为什么 Go 没有 try/catch？

这是另一个巨大争议。

很多语言：

Java：

```
try {

    readFile();

}

catch(Exception e){

}
```

------

Python：

```
try:

    open()

except:

    pass
```

------

Go：

```
file, err := os.Open("test.txt")

if err != nil {

    return err

}
```

------

很多新人：

第一反应：

"好麻烦。"

------

但是：

Go 的观点：

错误不是异常。

错误是程序正常运行的一部分。

------

# 第八节：异常机制的问题

假设：

一个函数：

```
funcA()
```

调用：

```
funcA

↓

funcB

↓

funcC

↓

funcD
```

其中：

funcD：

抛异常。

------

问题：

谁处理？

可能：

最上层。

------

于是：

错误处理流程：

隐藏起来。

------

代码阅读：

需要知道：

调用链。

------

Go：

错误：

显式传递。

------

例如：

```
result, err := funcB()

if err != nil {

    return err

}
```

你看到代码：

就知道：

这里可能失败。

------

# 第九节：显式的价值

大型系统里面：

最大的敌人：

不是代码多。

而是：

隐藏行为。

------

例如：

一个函数：

表面：

```
CreateUser()
```

但是内部：

偷偷：

- 发邮件
- 写日志
- 调第三方 API
- 修改库存

------

名字：

看不出来。

------

这就是：

隐藏副作用。

------

Go 喜欢：

明确。

------

代码：

应该尽量：

告诉读者真实发生什么。

------

# 第十节：为什么 Go 不喜欢魔法？

很多框架喜欢：

自动做很多事情。

例如：

自动扫描。

自动注入。

自动生成。

------

优点：

写起来快。

------

缺点：

出问题时：

不知道哪里发生。

------

Go 社区：

更偏向：

简单。

直接。

可调试。

------

例如：

Go 常见项目：

你打开代码。

基本能找到：

入口。

流程。

依赖。

------

# 第十一节：Go 的"少"其实是一种约束

这里有一个重要思想。

很多优秀工程：

都依赖约束。

------

例如：

Git。

为什么成功？

因为：

简单模型。

------

Unix。

为什么成功？

因为：

小工具组合。

------

Go。

为什么成功？

因为：

限制选择。

------

没有限制：

自由。

但是：

也带来混乱。

------

限制：

降低复杂度。

------

# 第十二节：Go 设计哲学总结

现在总结。

------

## 哲学一：

简单优于复杂。

不是：

功能越少越好。

而是：

减少不必要复杂度。

------

## 哲学二：

组合优于继承。

因为：

大型系统更需要灵活组合。

------

## 哲学三：

显式优于隐式。

让代码行为容易理解。

------

## 哲学四：

工具统一优于个人习惯。

统一：

降低团队成本。

------

# 企业案例

假设：

你设计一个审批流程系统。

需求：

以后可能增加：

- 请假审批
- 报销审批
- 采购审批
- 入职审批
- 离职审批

------

错误设计：

```
ApprovalBase

↓

LeaveApproval

↓

ExpenseApproval

↓

PurchaseApproval
```

几年以后：

继承树巨大。

------

Go 思想：

设计：

```
Approval

+

Rule

+

Condition

+

Action

+

Notification
```

不同流程：

组合不同能力。

------

这就是：

为什么 Go 的思想非常适合：

流程引擎。

------

# 本章总结

今天最重要的内容：

------

### 1

Go 的缺失不是不足。

是设计选择。

------

### 2

没有 class：

不是不能面向对象。

而是选择不同方式。

------

### 3

没有继承：

不是落后。

而是避免大型系统复杂关系。

------

### 4

没有异常：

不是错误处理弱。

而是强调错误透明。

------

### 5

Go 的核心：

降低长期维护成本。

------
