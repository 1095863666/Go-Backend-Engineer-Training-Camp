# 第一节：为什么需要同步？

看代码：

```
package main

import "fmt"


func main(){

    count:=0


    go func(){

        count++

    }()


    go func(){

        count++

    }()


    fmt.Println(count)

}
```

------

问题：

结果：

不确定。

可能：

```
0

1

2
```

------

为什么？

两个 Goroutine：

同时修改。

------

这叫：

**数据竞争（Data Race）**

------

# 第二节：什么是临界区？

临界区：

Critical Section。

------

指：

访问共享资源的代码。

------

例如：

```
count++
```

------

这个操作：

必须保证：

同一时间：

一个 Goroutine 执行。

------

# 第三节：Mutex 是什么？

Mutex：

Mutual Exclusion。

互斥锁。

------

简单理解：

一把门锁。

------

代码：

```
var mu sync.Mutex


mu.Lock()

count++

mu.Unlock()
```

------

流程：

```
Goroutine A

拿锁

执行

释放


Goroutine B

等待

拿锁

执行
```

------

# 第四节：Mutex 使用规则

必须：

成对出现：

```
Lock()

Unlock()
```

------

推荐：

```
mu.Lock()

defer mu.Unlock()
```

------

为什么？

防止：

中途 return。

------

例如：

```
func Update(){

    mu.Lock()

    defer mu.Unlock()


    if error{

        return

    }

}
```

------

即使 return：

也释放。

------

# 第五节：Mutex 常见错误

## 错误 1：忘记 Unlock

```
mu.Lock()

doSomething()
```

------

结果：

其他 Goroutine：

永久等待。

------

叫：

死锁。

------

# 第六节：什么是死锁？

Deadlock。

------

例如：

```
A 等待 B

B 等待 A
```

------

双方：

永远等待。

------

代码：

```
lockA.Lock()

lockB.Lock()
```

------

另一个：

```
lockB.Lock()

lockA.Lock()
```

------

可能：

死锁。

------

# 第七节：RWMutex

读写锁。

------

场景：

大量读取。

少量修改。

------

例如：

配置中心。

------

10000 个请求：

读取配置。

------

只有：

管理员修改一次。

------

Mutex：

所有读取排队。

------

RWMutex：

允许多个读。

------

# 第八节：RWMutex 基础

定义：

```
var rw sync.RWMutex
```

------

读：

```
rw.RLock()

value:=config

rw.RUnlock()
```

------

写：

```
rw.Lock()

config=value

rw.Unlock()
```

------

规则：

读：

多个同时。

------

写：

独占。

------

# 第九节：RWMutex 工作模型

例如：

5 个读：

```
Reader1 ┐

Reader2 ├── 可以同时

Reader3 ┘
```

------

写：

```
Writer

↓

所有停止
```

------

# 第十节：缓存设计案例

例如：

用户缓存：

```
type UserCache struct{

    mu sync.RWMutex

    data map[int]User

}
```

------

读取：

```
func(c *UserCache)Get(id int)User{

    c.mu.RLock()

    defer c.mu.RUnlock()


    return c.data[id]

}
```

------

更新：

```
func(c *UserCache)Set(
id int,
u User,
){

    c.mu.Lock()

    defer c.mu.Unlock()


    c.data[id]=u

}
```

------

这是企业常见写法。

------

# 第十一节：sync.Once

问题：

初始化。

例如：

数据库连接：

```
var db *DB
```

------

多个请求：

同时第一次访问。

------

可能：

创建多个连接。

------

需要：

只执行一次。

------

使用：

sync.Once。

------

# 第十二节：Once 示例

```
var once sync.Once


func Init(){

    once.Do(func(){

        connectDB()

    })

}
```

------

无论调用多少次：

只执行：

一次。

------

# 第十三节：Once 应用场景

常见：

## 单例

```
Config

Logger

Database
```

------

## 延迟初始化

lazy initialization。

------

例如：

第一次使用：

才加载。

------

# 第十四节：sync.Pool

什么问题？

对象频繁创建。

------

例如：

HTTP 请求：

每次：

创建 buffer。

------

大量：

短生命周期对象。

------

GC 压力大。

------

Pool：

对象复用。

------

# 第十五节：Pool 示例

```
var pool sync.Pool


pool.New = func()any{

    return make([]byte,1024)

}
```

------

获取：

```
buf:=pool.Get()
```

------

归还：

```
pool.Put(buf)
```

------

------

# 第十六节：Pool 工作思想

没有：

直接创建。

------

先：

从池拿。

------

使用。

------

放回。

------

减少：

GC。

------

# 第十七节：Pool 注意事项

不要：

保存长期对象。

------

Pool：

适合：

临时对象。

------

例如：

buffer。

------

不适合：

业务数据。

------

# 第十八节：Atomic 原子操作

有些情况：

不用锁。

------

例如：

计数器。

------

代码：

```
atomic.AddInt64(
&count,
1,
)
```

------

特点：

硬件级保证。

------

比 Mutex：

轻。

------

# 第十九节：Atomic 示例

统计请求数量：

```
var total int64


func Request(){

    atomic.AddInt64(
        &total,
        1,
    )

}
```

------

高并发：

非常常见。

------

# 第二十节：Mutex vs Atomic

| 方式   | 适合     |
| ------ | -------- |
| Mutex  | 复杂数据 |
| Atomic | 简单变量 |

------

例如：

计数：

Atomic。

------

Map：

Mutex。

------

# 第二十一节：Race Detector

Go 提供：

检测数据竞争。

------

运行：

```
go run -race main.go
```

------

例如：

发现：

```
WARNING: DATA RACE
```

------

开发阶段：

非常重要。

------

# 第二十二节：Flow Engine 并发设计

流程状态：

例如：

```
type FlowInstance struct{

    ID string

    Status string

}
```

------

多个 Worker：

可能：

同时更新。

------

错误：

```
instance.Status="RUNNING"
```

------

需要：

保护。

------

设计：

```
type FlowState struct{

    mu sync.Mutex

    Status string

}
```

------

更新：

```
func(s *FlowState)Update(
status string,
){

    s.mu.Lock()

    defer s.mu.Unlock()


    s.Status=status

}
```

------

# 第二十四节：什么时候用 Mutex？

```
Cache

Map
```

# 第二十五节：企业选择原则

不要：

就 Channel。

------

判断：

## 数据所有权转移

Channel。

------

使用：

Mutex。
