# 第一节：什么是并发？

很多人：

认为：

并发 = 同时执行。

这是错误的。

------

并发：

Concurrency。

意思：

> 一个程序同时管理多个任务。

------

例如：

餐厅：

一个服务员：

同时：

接待多个客户。

------

服务员：

不是同时炒十道菜。

------

但是：

他管理：

多个订单。

------

这就是：

并发。

------

# 第二节：什么是并行？

并行：

Parallelism。

是真正：

同时执行。

------

例如：

4 个厨师：

同时：

炒 4 道菜。

------

需要：

多个 CPU 核心。

------

区别：

| 概念 | 含义             |
| ---- | ---------------- |
| 并发 | 同时处理多个任务 |
| 并行 | 同时执行多个任务 |

------

# 第三节：传统线程模型

例如：

Java。

通常：

一个任务：

一个线程。

------

模型：

```
Request A

    Thread 1


Request B

    Thread 2


Request C

    Thread 3
```

------

问题：

线程：

比较重。

------

# 第四节：线程为什么重？

操作系统线程：

包含：

- 栈空间
- 调度信息
- 上下文切换

------

默认：

可能：

MB 级内存。

------

如果：

10 万任务。

------

线程：

成本巨大。

------

# 第五节：Goroutine 是什么？

Goroutine：

Go 创建的轻量级执行单元。

------

使用：

非常简单：

```
go task()
```

------

例如：

```
package main


import "fmt"


func hello(){

    fmt.Println("hello")

}


func main(){

    go hello()

}
```

------

`go`

表示：

启动一个 Goroutine。

------

# 第六节：Goroutine 和 Thread 区别

线程：

OS 管理。

------

Goroutine：

Go Runtime 管理。

------

结构：

线程：

```
OS

↓

Thread

↓

Code
```

------

Go：

```
OS Thread

↓

Go Scheduler

↓

Goroutine
```

------

------

# 第七节：为什么 Goroutine 很轻？

因为：

初始栈：

很小。

------

传统线程：

固定栈。

------

Goroutine：

动态增长。

------

所以：

可以：

创建：

几十万。

------

# 第八节：Goroutine 生命周期

例如：

```
go process()
```

过程：

```
创建 G

↓

放入调度队列

↓

Scheduler 分配执行

↓

完成

↓

销毁
```

------

------

# 第九节：为什么刚才代码没有输出？

代码：

```
func main(){

    go hello()

}
```

可能：

没有输出。

------

原因：

main 结束。

整个程序退出。

------

Goroutine：

也结束。

------

需要：

等待。

------

# 第十节：使用 Sleep 等待（不推荐）

例如：

```
func main(){

    go hello()

    time.Sleep(time.Second)

}
```

------

可以。

但是：

不是工程方案。

------

因为：

不知道：

任务多久完成。

------

# 第十一节：正确方式 sync.WaitGroup

例如：

```
var wg sync.WaitGroup


wg.Add(1)


go func(){

    defer wg.Done()

    work()

}()


wg.Wait()
```

------

解释：

Add：

增加任务数量。

------

Done：

完成一个。

------

Wait：

等待全部结束。

------

# 第十二节：WaitGroup 原理

内部：

维护：

计数器。

------

例如：

```
Add(3)

count=3


完成1个

count=2


完成2个

count=1


完成3个

count=0


释放 Wait
```

------

# 第十三节：并发安全问题

现在：

多个 Goroutine：

访问：

同一个变量。

------

例如：

```
count := 0


go func(){

    count++

}()


go func(){

    count++

}()
```

------

问题：

数据竞争。

------

# 第十四节：什么是 Race Condition？

例如：

count++：

不是一步。

实际上：

```
读取 count

↓

加1

↓

写回
```

------

两个 Goroutine：

同时：

读取。

------

结果：

错误。

------

# 第十五节：Mutex 解决

互斥锁。

------

例如：

```
var mu sync.Mutex


mu.Lock()

count++

mu.Unlock()
```

------

意思：

同一时间：

只有一个执行。

------

# 第十六节：Go 的并发思想

Go 推崇：

CSP。

------

Communicating Sequential Processes。

------

核心：

> 不要通过共享内存通信，而通过通信共享内存。

------

也就是：

不要：

大家修改同一个变量。

------

而是：

发送消息。

------

工具：

Channel。

------

# 第十七节：Channel 是什么？

Channel：

Goroutine 通信管道。

------

例如：

```
ch := make(chan int)
```

------

发送：

```
ch <- 100
```

------

接收：

```
value := <-ch
```

------

------

# 第十八节：简单 Channel 示例

```
func worker(ch chan int){

    ch <- 100

}


func main(){

    ch:=make(chan int)


    go worker(ch)


    result:=<-ch


    fmt.Println(result)

}
```

------

流程：

```
worker

↓

channel

↓

main
```

------

------

# 第十九节：Channel 为什么重要？

因为：

多个任务：

可以：

安全交换数据。

------

例如：

生产者：

```
生成任务
```

↓

channel

↓

消费者：

```
执行任务
```

------

这就是：

Worker Pool。

------

# 第二十节：Flow Engine 实战

流程引擎：

天然适合 Goroutine。

例如：

执行节点：

```
go executor.Execute()
```

------

多个流程：

同时：

运行。

------

结构：

```
Flow1

 ├ Node1

 ├ Node2


Flow2

 ├ Node1

 ├ Node2
```

------

每个：

可以：

独立 Goroutine。

------

# 第二十一节：但是流程引擎不能无限创建

错误：

```
for {

    go executeFlow()

}
```

------

结果：

Goroutine 爆炸。

------

需要：

限制。

------

方法：

Worker Pool。

------

# 第二十二节：Worker Pool 思想

例如：

100 个任务。

但是：

只允许：

10 个 worker。

------

结构：

```
Task Queue


 ↓


Worker1

Worker2

Worker3


 ↓


Result
```
