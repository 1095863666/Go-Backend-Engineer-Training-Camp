# iota——Go 为什么没有 enum？

------

> **一句话总结本章**

> **iota 本质不是一个关键字，而是 Go 提供的一种编译期"自动编号机制"，它比传统 enum 更简单、更灵活。**

------

# 本章学习目标

完成本章后，你应该能够回答：

- 什么是 `iota`？
- 为什么 Go 没有 `enum`？
- `iota` 为什么每一行都会自动加一？
- `iota` 如何定义状态？
- `iota` 如何定义位标志（Flag）？
- 企业项目什么时候适合使用 `iota`？

------

# 第一节：为什么需要编号？

假设你正在开发流程引擎。

节点状态：

```
Ready

Running

Success

Failed
```

计算机不知道：

什么叫：

Running。

所以：

需要数字表示。

例如：

```
Ready     -> 0

Running   -> 1

Success   -> 2

Failed    -> 3
```

这就是：

**状态编码。**

------

# 第二节：不用 iota 怎么写？

最直接：

```
const (
    Ready   = 0
    Running = 1
    Success = 2
    Failed  = 3
)
```

没有问题。

但是。

以后：

增加：

Pause。

变成：

```
const (
    Ready   = 0
    Running = 1
    Pause   = 2
    Success = 3
    Failed  = 4
)
```

你需要：

修改：

后面：

所有数字。

------

大型项目。

容易：

改错。

------

于是。

Go 提供：

自动编号。

------

# 第三节：什么是 iota？

写法：

```
const (
    Ready = iota
    Running
    Success
    Failed
)
```

编译器：

自动：

展开。

等价：

```
const (
    Ready   = 0
    Running = 1
    Success = 2
    Failed  = 3
)
```

注意：

不是运行时。

而是：

**编译时。**

------

# 第四节：iota 到底是什么？

很多书：

说：

> iota 是常量生成器。

这句话：

不准确。

更准确：

> **iota 是 const 代码块里的行号计数器。**

例如：

```
const (
    A = iota
    B
    C
)
```

可以理解成：

```
第一行：

iota = 0

第二行：

iota = 1

第三行：

iota = 2
```

所以：

得到：

```
A = 0

B = 1

C = 2
```

------

# 第五节：每进入一个 const，都会重新开始

例如：

```
const (
    A = iota
    B
)
```

得到：

```
A = 0

B = 1
```

下面：

再来：

```
const (
    C = iota
    D
)
```

注意。

不是：

```
2

3
```

而是：

重新：

```
C = 0

D = 1
```

因为：

每一个：

const 块。

都是：

新的计数器。

------

# 第六节：为什么 Go 不设计 enum？

这是重点。

例如：

Java：

```
enum Status {
    Ready,
    Running,
    Success
}
```

非常漂亮。

Go 为什么不要？

原因：

Go 希望：

**状态就是普通常量。**

没有：

特殊对象。

没有：

隐藏行为。

没有：

复杂语法。

------

Go：

只有：

```
const (
    Ready = iota
    Running
)
```

简单。

统一。

编译器容易处理。

程序员容易理解。

------

# 第七节：iota 的表达式

真正厉害的是：

它不仅能：

+1。

例如：

```
const (
    KB = 1 << (10 * iota)
    MB
    GB
    TB
)
```

编译结果：

```
KB = 1024

MB = 1048576

GB = 1073741824

TB = ...
```

这就是：

Go 标准库常见写法。

例如：

定义：

文件大小。

------

# 第八节：企业中的状态设计

流程引擎：

例如：

```
type NodeStatus int

const (
    NodeReady NodeStatus = iota
    NodeRunning
    NodeSuccess
    NodeFailed
    NodeCanceled
)
```

以后：

数据库：

保存：

```
0

1

2

3
```

代码：

看到：

```
NodeRunning
```

非常清晰。

------

# 第九节：为什么最好自定义类型？

很多新人：

这样写：

```
const (
    Ready = iota
    Running
)
```

其实：

可以。

但是：

更推荐：

```
type Status int

const (
    Ready Status = iota
    Running
)
```

为什么？

因为：

类型：

更安全。

例如：

```
type Color int

type Status int
```

虽然：

底层：

都是：

int。

但是：

语义：

完全不同。

这就是：

**类型表达业务。**

以后讲类型系统。

会重点分析。

------

# 第十节：iota 的限制

不要什么都用 iota。

例如：

HTTP：

```
200

404

500
```

不能：

```
const (
    OK = iota
)
```

因为：

HTTP 标准：

就是：

200。

不能：

自己编号。

------

也不要：

数据库：

状态：

如果：

数据库：

规定：

```
100

200

300
```

你就应该：

保持一致。

不要：

为了用：

iota。

而：

修改协议。

------

# 第十一节：Flow Engine 实战

假设：

我们的流程节点：

```

Success

Failed

Canceled
```

建议：

这样定义：

```
type NodeStatus int

const (
    NodeReady NodeStatus = iota
    NodeRunning
    NodeSuccess
    NodeFailed
    NodeTimeout
    NodeCanceled
)
```

以后：

判断：

```
if node.Status == NodeRunning {

}
```

代码：

一眼：

知道：

什么意思。

------

# 第十二节：什么时候使用 iota？

建议：

使用：

- 流程状态
- 权限状态
- 节点类型
- 日志等级
- 工作模式
- 配置枚举

不要使用：

- HTTP 状态码
- SQL 错误码
- 第三方协议编号
- 国家代码
- 用户 ID

原则：

**只有当编号由你自己定义时，才适合使用 iota。**
