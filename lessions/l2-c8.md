# make 与 new——Go 为什么设计了两个创建对象的方法？

------

> **一句话总结本章**

> **new 负责申请内存；make 负责初始化 Go 内置的引用类型。**

很多人背过这句话。

但是。

真正理解它，需要先理解一个问题。

------

# 第一节：什么叫"创建对象"？

来看：

```
var age int
```

发生了什么？

其实只有一件事。

Go：

申请了一块内存。

例如：

```
0x1000

↓

0
```

然后：

变量：

```
age
```

指向：

这块内存。

------

那么。

创建对象：

到底意味着什么？

实际上包含：

两步。

```
第一步：

申请内存

↓

第二步：

初始化对象
```

很多语言：

把这两步：

合在一起。

Go：

故意分开。

为什么？

后面马上解释。

------

# 第二节：什么是 new？

来看：

```
p := new(int)
```

很多人：

第一次看到：

就懵了。

------

真正发生：

```
申请：

4（或8）字节

↓

默认值：

0

↓

返回地址
```

所以：

```
fmt.Println(*p)
```

输出：

```
0
```

注意。

返回的是：

```
*int
```

不是：

```
int
```

------

为什么？

因为：

`new`

返回：

指针。

------

# 第三节：为什么返回指针？

假设：

不是：

返回：

地址。

而是：

返回：

值。

例如：

```
age := new(int)
```

得到：

```
0
```

那么：

刚才：

申请的内存：

还有什么意义？

没有。

所以：

必须：

返回：

地址。

让程序：

以后：

还能访问。

------

# 第四节：new 能创建任何类型吗？

例如：

```
new(int)
```

可以。

------

```
new(User)
```

可以。

------

```
new([100]int)
```

可以。

------

因为：

new：

根本：

不知道：

业务。

它只负责：

申请内存。

------

# 第五节：为什么 new 不适合 Slice？

来看：

```
s := new([]int)
```

很多新人：

认为：

已经：

创建：

Slice。

其实：

没有。

看看：

```
fmt.Println(*s)
```

输出：

```
[]
```

但是。

注意。

里面：

没有：

底层数组。

为什么？

因为：

Slice：

不仅需要：

Slice Header。

还需要：

真正的数据区域。

------

new：

只负责：

Header。

不会：

创建：

底层数组。

------

所以：

不能直接：

使用。

------

# 第六节：Slice 真正需要什么？

回忆：

上一章。

Slice：

内部：

可以理解成：

```
type slice struct{

    ptr

    len

    cap
}
```

如果：

只有：

这个：

Header。

没有：

数组。

例如：

```
ptr=nil

len=0

cap=0
```

还能：

append。

可以。

但是：

第一次：

一定：

重新分配。

------

Go：

为了方便。

设计：

make。

------

# 第七节：make 到底做了什么？

例如：

```
s := make([]int,5)
```

真正发生：

不是：

一步。

而是：

四步。

```
申请：

Slice Header

↓

申请：

底层数组

↓

Header：

指向数组

↓

返回 Slice
```

注意。

这里：

已经：

初始化完成。

所以：

可以：

立即使用。

------

# 第八节：为什么叫 make？

很多语言：

叫：

Create。

New。

Init。

Go：

为什么：

叫：

make？

因为：

它不仅：

申请。

还要：

把：

对象：

"做好"。

例如：

Slice。

Map。

Channel。

这些：

都不是：

一块内存。

它们：

都有：

运行时结构。

------

所以：

Go：

不用：

new。

而：

make。

------

# 第九节：Map 也是一样

例如：

```
var m map[string]int
```

注意。

现在：

Map：

还是：

nil。

执行：

```
m["Tom"] = 100
```

直接：

panic。

为什么？

因为：

Hash Table：

还没：

创建。

------

正确：

```
m := make(map[string]int)
```

这样：

Hash Bucket：

初始化。

才能：

存数据。

------

# 第十节：Channel 也是一样

例如：

```
var ch chan int
```

现在：

还是：

nil。

发送：

```
ch <- 1
```

永久阻塞。

------

正确：

```
ch := make(chan int)
```

Go：

创建：

Channel：

内部结构。

才能：

通信。

------

# 第十一节：为什么 make 不能创建 int？

例如：

```
make(int)
```

编译：

失败。

为什么？

因为：

int：

没有：

初始化逻辑。

只需要：

申请：

内存。

------

make：

只负责：

复杂对象。

------

# 第十二节：企业实践

以后：

看到：

下面：

直接：

想到：

make。

```

```

------

看到：

下面：

想到：

new。

```

```

当然。

企业：

更多时候：

甚至：

直接：

```

```

这个。

后面：

结构体章节。

详细分析。

------

# 第十三节：为什么企业很少写 new？

来看：

```
user := new(User)
```

很多公司：

更喜欢：

```
user := &User{}
```

为什么？

因为：

第二种：

可以：

直接：

初始化。

例如：

```
user := &User{

    Name:"Tom",

    Age:18,

}
```

所以。

真正企业：

`new` 的使用频率：

没有你想象中那么高。

但是：

理解它：

非常重要。

------

# 第十四节：Flow Engine 实战

流程引擎：

例如：

执行节点：

```
type ExecuteContext struct{

    Variables map[string]any

    Nodes []Node

}
```

初始化：

```
ctx := ExecuteContext{

    Variables: make(map[string]any),

    Nodes: make([]Node,0,100),

}
```

注意。

这里：

全部：

使用：

make。

因为：

Map。

Slice。

都需要：

初始化。

# 作者的话（课程重大调整）

到这里，我决定正式调整整个课程结构。

我认为，目前最大的不足是：

**我们已经理解了 Go 的设计思想，但还没有足够的"动手练习"。**

所以从 **Lesson 3** 开始，每一课都会固定增加两个模块：

1. Lab（实验）
   - 动手验证今天学到的知识。
   - 例如观察 `append` 前后内存地址是否变化。
2. Debug（调试）
   - 给你一段有 Bug 的代码。
   - 自己定位问题。
   - 学会阅读错误，而不是只会写代码。

这样，你不仅知道 Go 是怎么设计的，还会真正具备解决问题的能力。

## 我对课程还有一个建议

如果我们的目标是 **100 节左右的 Go 后端课程**，我建议后续采用如下节奏：

- **一章理论（Language Design）**
- **一章实验（Lab）**
- **一章企业实践（Project）**
- **一章源码阅读（Source Dive）**
- **一章综合练习（Challenge）**

例如学完 Slice，不是立刻进入下一个语法，而是：

- Slice 实验：验证 `append` 扩容。
- Slice 企业案例：为什么 Gin、GORM 大量使用 Slice。
- Slice 源码：阅读 `runtime.growslice`。
- Slice 挑战题：分析真实 Bug。

这样学完 100 节后，你掌握的不是"Go 语法"，而是真正的 **Go 工程能力**。我认为这会比市面上的大多数 Go 教程更加系统，也更接近企业实际开发。
