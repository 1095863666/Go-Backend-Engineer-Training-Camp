# Method 深入——receiver、方法集与接口实现隐藏规则

------

> **一句话总结本章**

> **Go 判断一个类型是否实现接口，不看你有没有“写过方法”，而看这个类型的方法集合是否满足接口要求。**

------

# 本章学习目标

完成本章后，你应该能够：

- 理解 method receiver
- 理解方法集（Method Set）
- 理解 T 和 *T 的区别
- 理解为什么 pointer receiver 影响接口实现
- 理解自动取地址规则
- 正确设计接口实现

------

# 第一节：什么是 Method？

上一章：

我们写过：

```
type User struct{

    Name string

}
```

添加：

```
func (u User) Say(){

}
```

这个：

叫：

方法。

------

函数：

属于 package。

例如：

```
func Save()
```

------

方法：

属于类型。

例如：

```
user.Save()
```

------

# 第二节：receiver 有两种形式

Go 方法：

两种 receiver。

------

## 第一种：值 receiver

```
func (u User) Name(){

}
```

------

## 第二种：指针 receiver

```
func (u *User) Name(){

}
```

------

区别：

非常重要。

------

# 第三节：值 receiver 发生什么？

例如：

```
type User struct{

    Name string

}


func (u User) Print(){

}
```

------

调用：

```
user.Print()
```

实际上：

类似：

```
User.Print(user)
```

------

传进去：

是复制。

------

所以：

方法里面：

修改：

不会影响外部。

------

# 第四节：指针 receiver 发生什么？

例如：

```
func (u *User) SetName(){

}
```

调用：

```
user.SetName()
```

Go：

自动：

转换：

```
(&user).SetName()
```

------

所以：

可以修改。

------

# 第五节：什么是方法集？

这是本章重点。

简单理解：

> 一个类型拥有的方法列表。

------

例如：

```
type User struct{}
```

有：

```
func (User) A(){

}
```

那么：

User：

拥有：

A。

------

# 第六节：T 和 *T 的方法集不同

这是核心规则。

假设：

```
type User struct{}
```

------

定义：

```
func (u User) Read(){

}
```

------

那么：

两个类型：

都有：

Read。

------

也就是：

```
User

Read


*User

Read
```

------

但是：

如果：

```
func (u *User) Write(){

}
```

情况：

不同。

------

现在：

```
User

没有 Write


*User

有 Write
```

------

记住：

> 指针 receiver 方法，只属于指针类型。

------

# 第七节：为什么会这样？

因为：

值：

不知道：

是否可以安全取地址。

------

例如：

```
User{}.Write()
```

这个：

临时对象。

没有地址。

------

所以：

Go 不允许。

------

# 第八节：接口实现规则

现在：

定义接口：

```
type Saver interface{

    Save()

}
```

------

情况 1：

值 receiver：

```
func (u User) Save(){

}
```

那么：

```
User      实现 Saver

*User     实现 Saver
```

------

情况 2：

指针 receiver：

```
func (u *User) Save(){

}
```

那么：

```
User      不实现 Saver


*User     实现 Saver
```

------

这是面试高频。

------

# 第九节：经典错误

代码：

```
package main


type Saver interface{

    Save()

}


type User struct{}


func (u *User) Save(){

}


func main(){

    var s Saver

    s = User{}

}
```

错误：

类似：

```
User does not implement Saver
(method Save has pointer receiver)
```

------

为什么？

因为：

User：

方法集：

没有：

Save。

------

只有：

*User：

有。

------

# 第十节：正确方式

方式 1：

使用指针：

```
s = &User{}
```

------

方式 2：

改成值 receiver：

```
func (u User) Save(){

}
```

------

选择：

取决于：

设计。

------

# 第十一节：为什么企业代码大量使用 pointer receiver？

看：

数据库模型。

例如：

```
type User struct{

    ID int

    Name string

}
```

------

方法：

```
func (u *User) Update(){

}
```

------

原因：

1. ​

修改对象。

1. ​

避免复制。

1. ​

保持一致。

------

所以：

业务 struct：

通常：

默认：

pointer receiver。

------

# 第十二节：为什么标准库很多值 receiver？

例如：

time.Time。

为什么？

因为：

它是：

不可变类型。

------

值复制：

成本低。

并且：

安全。

------

所以：

不是：

指针一定好。

------

# 第十三节：receiver 选择原则

记忆：

------

## 使用值 receiver

如果：

- struct 很小
- 不修改状态
- 希望类型值和指针都实现接口

例如：

```
type Money struct{

    Value int

}
```

------

## 使用指针 receiver

如果：

- 修改数据
- struct 较大
- 包含 mutex
- 生命周期需要统一

例如：

```
type Service struct{

}
```

------

# 第十四节：mutex 特殊情况

例如：

```
type Counter struct{

    sync.Mutex

    Value int

}
```

不要：

复制。

为什么？

因为：

锁复制：

非常危险。

------

所以：

一定：

pointer receiver。

------

# 第十五节：Flow Engine 实战

我们的节点执行：

定义：

```
type NodeExecutor interface{

    Execute() error

}
```

------

节点：

```
type SQLNode struct{

    SQL string

}
```

------

执行：

```
func (n *SQLNode) Execute() error{

    return nil

}
```

------

为什么：

使用：

`*SQLNode`？

因为：

执行过程：

可能修改：

状态。

例如：

```
n.Status = Running
```

------

所以：

接口存：

```
var executor NodeExecutor

executor=&SQLNode{}
```

------

不要：

```
executor=SQLNode{}
```

------

# 第十六节：接口设计中的隐藏影响

假设：

以后：

有人写：

```
func TestExecutor(e NodeExecutor){

}
```

------

如果你的实现：

使用：

pointer receiver。

那么：

调用者：

必须：

传：

指针。

------

所以：

接口设计：

会影响：

整个系统。

------

# 第十七节：方法集总结表

非常重要。

| 方法            | User | *User |
| --------------- | ---- | ----- |
| func(User) A()  | ✔    | ✔     |
| func(*User) B() | ❌    | ✔     |

------

接口判断：

看：

左边。

------

# 第十八节：企业代码建议

大型 Go 项目：

常见：

模式：

```
type Service struct{

}


func (s *Service) Create(){

}


func (s *Service) Update(){

}
```

------

然后：

接口：

```
type UserService interface{

    Create()

    Update()

}
```

------

实现：

```
var _ UserService = (*Service)(nil)
```

------

这是什么？

编译检查。

------

意思：

> 如果 Service 没实现接口，编译直接失败。

------

这是企业常用技巧。
