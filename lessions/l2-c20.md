# Channel 深入——Go 并发通信的核心机制

------

> **一句话总结本章**

> **Channel 是 Goroutine 之间传递数据的安全通道，它把“共享内存竞争”转换成“消息传递”。**

------

# 本章学习目标

完成本章后，你应该能够：

- 理解 Channel 本质
- 理解无缓冲 Channel
- 理解有缓冲 Channel
- 理解发送和接收阻塞
- 理解 close
- 使用 range 遍历 Channel
- 掌握 select
- 实现 Worker Pool
- 设计 Flow Engine 调度通信模型

------

# 第一节：为什么需要 Channel？

上一章：

我们有：

```
count++
```

多个 Goroutine：

同时修改。

问题：

数据竞争。

------

传统方式：

加锁。

例如：

```
mutex.Lock()

data++

mutex.Unlock()
```

------

但是：

Go 提供另一种思想：

不要共享数据。

------

改成：

传递数据。

------

例如：

不要：

```
Worker A

修改

共享变量
```

------

改成：

```
Worker A

↓

Channel

↓

Worker B
```

------

# 第二节：Channel 是什么？

定义：

```
ch := make(chan int)
```

创建：

一个 int 类型通道。

------

可以理解：

像一个队列。

------

发送：

```
ch <- 100
```

------

接收：

```
x := <-ch
```

------

结构：

```
        Channel

      +---------+

100 ->|         |-> receive

      +---------+
```

------

# 第三节：最简单例子

代码：

```
package main

import "fmt"


func worker(ch chan int){

    ch <- 100

}


func main(){

    ch := make(chan int)


    go worker(ch)


    result := <-ch


    fmt.Println(result)

}
```

------

执行：

worker：

发送：

100。

------

main：

接收：

100。

------

输出：

```
100
```

------

# 第四节：Channel 默认是什么？

刚才：

```
make(chan int)
```

叫：

无缓冲 Channel。

------

特点：

发送：

必须等待接收。

------

接收：

必须等待发送。

------

也叫：

同步 Channel。

------

# 第五节：无缓冲 Channel 原理

例如：

```
ch <- 100
```

------

如果：

没有人接：

发送者等待。

------

过程：

```
Worker

发送100

  |
  |
等待
  |
  ↓

Main

接收100
```

------

发送和接收：

必须同时发生。

------

# 第六节：为什么设计成阻塞？

因为：

保证同步。

------

例如：

生产者：

生产数据。

消费者：

一定收到。

------

避免：

数据丢失。

------

# 第七节：有缓冲 Channel

创建：

```
ch := make(chan int,10)
```

------

表示：

容量：

10。

------

现在：

可以存：

10 个数据。

------

例如：

```
ch <- 1

ch <- 2

ch <- 3
```

------

不会阻塞。

------

# 第八节：有缓冲 Channel 结构

类似：

队列：

```
Channel

+---+---+---+

| 1 | 2 | 3 |

+---+---+---+
```

------

发送：

放进去。

------

接收：

取出来。

------

# 第九节：什么时候使用缓冲？

例如：

任务队列。

------

Flow Engine：

节点任务：

可能：

瞬间产生很多。

------

设计：

```
taskQueue := make(chan Task,1000)
```

------

生产者：

提交任务。

------

Worker：

消费。

------

# 第十节：Channel 发送阻塞

例如：

```
ch := make(chan int,2)


ch <- 1

ch <- 2

ch <- 3
```

------

第三次：

阻塞。

------

因为：

容量满了。

------

# 第十一节：Channel 接收阻塞

例如：

```
ch := make(chan int)


x := <-ch
```

------

没有发送：

永久等待。

------

所以：

生产者消费者必须匹配。

------

# 第十二节：close Channel

Channel：

可以关闭。

------

例如：

```
close(ch)
```

------

表示：

不会再发送数据。

------

注意：

关闭：

不是删除。

------

里面已有数据：

仍然可以读取。

------

# 第十三节：读取关闭后的 Channel

例如：

```
value, ok := <-ch
```

------

返回：

两个值。

------

正常：

```
value=100

ok=true
```

------

关闭：

```
value=0

ok=false
```

------

# 第十四节：为什么需要 ok？

判断：

数据是否结束。

------

例如：

消费者：

不知道：

什么时候停止。

------

通过：

close。

通知。

------

# 第十五节：range 遍历 Channel

非常常用。

例如：

```
for value := range ch {

    fmt.Println(value)

}
```

------

特点：

自动等待。

------

直到：

Channel close。

------

例如：

```
func producer(ch chan int){

    for i:=0;i<5;i++{

        ch <- i

    }

    close(ch)

}
```

------

消费者：

```
for v:=range ch{

}
```

------

收到：

0-4。

结束。

------

# 第十六节：谁应该 close Channel？

重要规则：

> 通常由发送方关闭 Channel。

------

为什么？

发送方知道：

什么时候结束。

------

消费者：

不知道。

------

错误：

消费者：

```
close(ch)
```

可能：

造成：

panic。

------

# 第十七节：关闭已关闭 Channel

错误：

```
close(ch)

close(ch)
```

------

结果：

panic。

------

所以：

close：

只能一次。

------

# 第十八节：select 是什么？

Channel 多路等待。

------

例如：

同时等待：

两个 Channel。

------

代码：

```
select {

case a := <-ch1:

    fmt.Println(a)


case b := <-ch2:

    fmt.Println(b)

}
```

------

哪个先完成：

执行哪个。

------

# 第十九节：select 应用场景

非常多。

------

## 超时控制

例如：

请求：

最多等待：

3 秒。

------

```
select {

case result := <-ch:

    return result


case <-time.After(time.Second*3):

    return timeout

}
```

------

# 第二十节：Flow Engine 超时设计

节点执行：

例如：

调用外部系统。

------

不能：

无限等待。

------

设计：

```
select {

case result:=<-nodeResult:

    save(result)


case <-timeout:

    markFailed()

}
```

------

# 第二十一节：Worker Pool

这是企业最常见模型。

------

需求：

10000 个任务。

但是：

最多：

10 个并发。

------

结构：

```
             Tasks

               |

               v

          +---------+

          | Channel |

          +---------+

          /    |    \

       W1     W2    W3
```

------

# 第二十二节：简单 Worker Pool

任务：

```
type Job struct{

    ID int

}
```

------

队列：

```
jobs := make(chan Job,100)
```

------

Worker：

```
func worker(id int,jobs chan Job){

    for job := range jobs{

        process(job)

    }

}
```

------

启动：

```
for i:=0;i<10;i++{

    go worker(i,jobs)

}
```

------

提交：

```
jobs <- Job{1}
```

------

关闭：

```
close(jobs)
```

------

# 第二十三节：Flow Engine 调度模型

一个真实流程：

例如：

```
开始

 |

节点A

 |

 +------+

 |      |

节点B  节点C

 |      |

 +------+

 |

结束
```

------

调度：

可以：

使用：

Channel。

------

任务：

```
type Task struct{

    NodeID string

    FlowID string

}
```

------

进入：

```
taskQueue <- task
```

------

Worker：

执行。

------

结果：

```
resultQueue <- result
```

# 第二十五节：Channel vs Mutex

什么时候用？

------

例如：

例如：

任务队列。

------

不是：

谁替代谁。
